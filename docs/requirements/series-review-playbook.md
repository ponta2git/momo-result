# 戦績比較 振り返り・次回プレイブック要求仕様

本書は、`docs/requirements/series-comparison.md` で定義済みの戦績比較ページに追加する「振り返り」機能の要求仕様である。

既存の試合確定、OCR、CSV/TSV出力、マスタ管理、戦績比較の基礎指標は変更しない。確定済み戦績データだけを使い、プレーヤーに対して次回以降の見どころや改善仮説を提示する。

---

## 0. 最終判断

MVPの推奨ライン:

- AI連係、AIによる自由文生成、外部推論エンジンはMVPに含めない。
- 既存の戦績比較ページ内に、分析サブページタブとして「振り返り」を追加する。別ページ化しない。
- 「振り返り」は分析タブの先頭に置く。ただし既存の詳細指標タブは根拠の正本として維持する。
- 「開催回レビュー」と「期間プレイブック」はアプリ全体の別モードにはしない。振り返りタブ内の上段・下段として並べる。
- 開催回レビューは、最新または指定した開催回の4戦前後を、選択中スコープの長期傾向と照合して読む。
- 期間プレイブックは、選択中の作品、シーズン、マップの集計から、各プレーヤーに最大3件の「次の一手」を出す。
- 既存指標を振り返りタブで再掲しない。振り返りタブは「解釈・示唆・根拠リンク」に集中する。
- 上部サマリーや既存 `highlights` と重複する話題性のある結論は、振り返りタブへ寄せる。既存タブ側は詳細根拠を読む場所にする。
- 推薦は決定論的に生成する。順位スコア、効果量、対象件数、縮小推定、信頼度、行動可能性を使い、サンプル不足による過剰な断定を避ける。
- 新しいプレーデータは取得しない。行動順、カード内容、目的地距離、移動距離、リアルタイム意思決定は推定しない。

## 1. 目的と提供価値

### 1.1 目的

既存の戦績比較ページは、平均順位、資産分布、収益、目的地、番手、スリの銀次などを詳細に比較できる。一方で、対戦後の振り返りでは、利用者が複数タブの数値を読んで「結局、次は何を意識すればよいか」へ変換する必要がある。

振り返り機能の目的は、既存指標を増やすことではなく、蓄積済み戦績から以下を短時間で得られるようにすることである。

- その開催回で何が起きたか。
- それが長期傾向と一致しているか、ズレているか。
- 各プレーヤーが次回の4戦で見るべきテーマは何か。
- そのテーマの根拠はどの指標か。

### 1.2 提供価値

| 観点 | 内容 |
|---|---|
| なぜ | 既存の詳細分析は有用だが、振り返り時に「次回へ持ち越す論点」へ読み替える負担がある。 |
| 何を行うか | 開催回4戦を長期傾向と照合し、各プレーヤーに最大3件の次回テーマを提示する。 |
| どのくらい | 20〜30分かかり得る雑談的な読み解きを、5分程度で論点化する。各自のテーマは1〜3件に絞る。 |
| どんな価値 | 勝ち筋の再現、崩れ条件の把握、次回の検証ポイント共有を支援する。 |

### 1.3 成功条件

- 利用者が振り返りタブを開いてから30秒以内に、対象開催回の主要論点と各プレーヤーの次回テーマを把握できる。
- 各プレーヤーに0〜3件のプレイブックカードが表示され、カードごとに「伸ばす」「修正する」「検証する」のいずれかが分かる。
- すべてのカードに、根拠指標、対象件数、信頼度、詳細タブへのリンクがある。
- 対象件数が少ない条件を、主提案として断定しない。
- 振り返りタブ内で既存指標の表やグラフを重複表示しない。
- 既存タブへ移動すれば、カードの根拠となる詳細指標を確認できる。

## 2. 用語

| 用語 | 意味 |
|---|---|
| 振り返り | 戦績比較ページに追加する分析サブページ。開催回レビューと期間プレイブックを含む。 |
| 開催回レビュー | 最新または指定した `held_events` の試合を、選択スコープの長期傾向と比較して読む要約。 |
| 期間プレイブック | 選択中の作品、シーズン、マップの確定済み戦績から作る、プレーヤー別の次回テーマ。 |
| 次の一手 | 次回以降の対戦で注目する具体的なテーマ。リアルタイムの手番戦略ではなく、振り返り後の観察・改善仮説を指す。 |
| ベースライン | 期間プレイブックと開催回レビューの比較基準にする確定済み戦績の集合。原則として画面で選択中の作品、シーズン、マップスコープを使う。 |
| 根拠チップ | プレイブックカードに付ける短い根拠表示。指標名、対象件数、信頼度、詳細リンクを含む。 |

## 3. データ前提と制約

### 3.1 利用できるデータ

以下の既存データだけを使う。

- `matches` の確定済み試合。
- `held_events`、`played_at`、`match_no_in_event`。
- 作品、シーズン、マップ。
- 固定4名のプレー順、順位、総資産、収益額。
- 固定6事件: 目的地、プラス駅、マイナス駅、カード駅、カード売り場、スリの銀次。

### 3.2 利用しない・推定しないデータ

以下はMVPで扱わない。

- 行動順。
- カードの種類、購入、使用内容。
- 目的地までの距離。
- 移動距離、駅数、マス数。
- ターン単位の状況。
- リアルタイムの意思決定。
- 新しいプレーデータ入力。
- AIまたは外部推論エンジン。

### 3.3 データ量の設計前提

公開仕様には本番DBの実測値を固定値として置かない。設計上は、作品単位では数十〜百数十戦規模の履歴があり、作品全体の傾向は一定の信頼度で扱える前提とする。

ただし、シーズン、マップ、条件付き指標で細かく絞ると対象件数は急に減る。そのため、プレイブックの採用可否は件数と信頼度で制御する。

## 4. IA / UX要求

### 4.1 配置

戦績比較ページの分析サブページタブを以下の並びへ拡張する。

1. 振り返り
2. 順位と相性
3. 勝ち筋
4. 流れと勢い
5. 番手と出来事

「振り返り」は結論と示唆を読む場所とし、既存4タブは根拠の詳細を読む場所とする。

### 4.2 別ページ化しない理由

振り返りは戦績比較ページの派生ではなく上位解釈レイヤーである。別ページ化すると、作品、シーズン、マップ選択、認証、URL共有、根拠指標への往復が重くなる。MVPでは既存ページ内タブとして実装する。

### 4.3 強いモード分割にしない理由

開催回レビューは直近4戦前後を扱うため、単独では推薦根拠として弱い。期間プレイブックは長期傾向を扱えるが、開催後の会話から遠くなる。

そのため、画面全体を「開催回モード」「期間モード」に切り替えるのではなく、振り返りタブ内で以下を上下に並べる。

1. 今回のレビュー: 対象開催回で何が起きたか。
2. 期間プレイブック: 蓄積傾向から次回何を見るか。

### 4.4 画面構造

振り返りタブは以下の順に構成する。

1. レビュー対象
   - 初期値は選択スコープ内で確定済み対戦が最も新しい開催回。
   - 過去開催回を選べる。
   - ベースラインがどのスコープかを明示する。
2. 今回の3行まとめ
   - 今日出た勝ち筋。
   - 今日の荒れ要因。
   - 次回に持ち越す論点。
3. 開催回レビュー
   - 第1〜第4試合を中心に、勝者、資産差、物件収益トップ、スリの銀次、接戦、大差などを見る。
   - 第5試合以降がある場合は補助表示にする。
   - 長期傾向と一致した点、ズレた点を分ける。
4. 個人別プレイブック
   - プレーヤーごとに最大3件。
   - 「伸ばす」「修正する」「検証する」の分類を表示する。
   - 根拠チップと詳細リンクを付ける。
5. 根拠リンク
   - カードごとに既存タブの該当ブロックへ移動できる導線を出す。

### 4.5 URL状態

既存の `view` query に `review` を追加する。

- `view=review` は振り返りタブを表す。
- 振り返り拡張実装後、URL query がない場合は `review` を初期表示にする。
- 既存の直接リンク互換のため、`view=overview`、`view=drivers`、`view=flow`、`view=context` は維持する。
- `view` が不正値の場合は `review` に戻す。
- 開催回選択は `reviewHeldEventId` query で復元できるようにする。
- `reviewHeldEventId` は表示・レビュー対象の状態であり、期間プレイブックのベースラインスコープを変えない。

## 6. 既存指標との関係

### 6.1 役割分担

| 領域 | 役割 |
|---|---|
| 振り返り | 既存指標を解釈し、開催回の論点と次回テーマへ変換する。 |
| 順位と相性 | 平均順位、順位分布、直接対決、順位ブレの詳細根拠。 |
| 勝ち筋 | 資産分布、物件収益、目的地、勝ち筋分類の詳細根拠。 |
| 流れと勢い | 荒れ試合、直近8戦、開催回内第n試合の詳細根拠。 |
| 番手と出来事 | 番手、カード売り場と目的地、スリの銀次の詳細根拠。 |

### 6.2 重複回避

- 振り返りタブでは、詳細指標の表やグラフを再掲しない。
- プレイブックカード内の数値は、根拠チップとして最小限にする。
- 詳細を読みたい場合は既存タブへ移動させる。
- 既存の上部主要サマリーは、振り返り拡張時に縮小を検討する。
- 上部に残す値は、対象スコープ、対戦数、データ信頼度、振り返り導線など、ページ全体の状態把握に寄せる。
- 首位社長、スリの銀次、平均順位差など話題性の強いサマリーは、振り返りタブの「今回の3行まとめ」または「開催回レビュー」へ移す。

### 6.3 既存指標を落とす条件

既存指標は原則として削除しない。削除または移動を検討するのは、以下をすべて満たす場合に限る。

- 振り返りタブと既存タブで同じ対象、同じ分母、同じ表示粒度の値を出している。
- 既存タブ側で詳細根拠としての役割を失っている。
- 削除しても、プレイブックカードの根拠を追跡できる。

## 7. プレイブック生成ルール

### 7.1 基本方針

推薦は決定論的に生成する。AIや自由入力に依存しない。

カードは以下の問いに答える。

- なぜ今これを見るべきか。
- 何を伸ばす、修正する、または検証するのか。
- どのくらい根拠があるか。
- 次回4戦で何を観察すればよいか。

カードは行動断定ではなく、観察・改善仮説として提示する。

### 7.2 カード分類

| 分類 | 意味 | 例 |
|---|---|---|
| 伸ばす | すでに勝ちに効いている傾向を再現する。 | 低資産帯が少なく、安定して入賞できている。 |
| 修正する | 成績を落としやすい条件を確認する。 | 物件収益トップ時に勝ち切れていない。 |
| 検証する | 長期傾向と直近傾向がズレている、または件数が少ないため次回見たい仮説。 | 直近は目的地なしでも上位に入れているが、長期ではまだ不明。 |

### 7.3 候補カテゴリ

MVPの候補カテゴリ:

- 順位の地力: 平均順位、入賞率、下位率、順位ブレ。
- 勝ち筋: 資産分布、物件収益比率、物件収益トップ時の結果。
- 目的地: 目的地上位、目的地ゼロ、目的地下位の結果。
- 事故耐性: スリの銀次被害時の平均順位、被害時下位率。
- 番手: 得意番手、苦手番手、番手差。
- 流れ: 直近8戦、連続入賞、連続下位、開催回内第n試合。
- 相性: 直接対決、平均順位差、平均資産差。
- 資産差: 勝利時の資産水準、2位時の惜敗差、下位時の負け幅。

### 7.4 順位スコア

順位系の比較では、必要に応じて順位スコアを使う。

```text
rankScore = 5 - rank
```

1位=4点、2位=3点、3位=2点、4位=1点とする。高いほど良い。

### 7.5 効果量

候補ごとに、本人ベースラインまたは選択スコープ全体との差分を計算する。

例:

- 条件あり平均順位スコア - 本人平均順位スコア。
- 条件あり入賞率 - 本人入賞率。
- 条件あり下位率 - 本人下位率。
- 直近8戦平均順位スコア - 長期平均順位スコア。

効果量は表示上の断定に使う前に、対象件数と信頼度で調整する。

### 7.6 縮小推定

対象件数が少ない条件の極端な値を過大評価しないため、MVPでは単純な縮小推定を使ってよい。

順位スコア差の例:

```text
shrunkEffect = rawEffect * targetCount / (targetCount + priorWeight)
```

`priorWeight` は固定閾値として実装に集約する。初期値は8を目安にしてよいが、テストで固定し、将来調整可能にする。

率の例:

```text
shrunkRate = (successCount + priorWeight * baselineRate) / (targetCount + priorWeight)
```

### 7.7 信頼度

カード候補は以下の件数基準で扱う。

| 対象 | 主採用 | 参考 | 非表示 |
|---|---:|---:|---:|
| 通常指標 | 20戦以上 | 3〜19戦 | 2戦以下 |
| 条件付き指標 | 8件以上 | 3〜7件 | 2件以下 |

通常指標は平均順位、入賞率、順位ブレ、資産分布など、プレーヤーの全参加試合を分母にする指標を指す。

条件付き指標は、物件収益トップ時、目的地ゼロ時、スリの銀次被害時、特定番手、特定開催回内試合番号など、条件で対象を絞る指標を指す。

### 7.8 機会スコア

カード候補は以下の考え方で優先順位を付ける。

```text
opportunityScore = abs(shrunkEffect) * exposureWeight * reliability * actionability
```

| 要素 | 意味 |
|---|---|
| `shrunkEffect` | 縮小後の効果量。大きいほど示唆が強い。 |
| `exposureWeight` | その条件が発生する頻度。頻度が高いほど次回も観察しやすい。 |
| `reliability` | 件数と信頼度。件数不足なら下げる。 |
| `actionability` | 次回の意識や振り返りに使いやすい度合い。 |

`actionability` は主観で都度決めない。カテゴリごとの定数として定義する。

例:

- 物件収益トップ時の勝ち切り、目的地との付き合い方、低資産帯回避: 高。
- 番手差、スリの銀次被害耐性: 中。
- 直接対決、開催回内第n試合: 中または参考。
- 偶発性が強い単発イベント: 低。

### 7.9 カード選定

プレーヤーごとに以下の順で選ぶ。

1. 候補カテゴリごとに最も強いカード候補を作る。
2. 非表示条件に該当する候補を除外する。
3. `opportunityScore` で並べる。
4. 同じカテゴリから複数出さない。
5. 可能なら「伸ばす」「修正する」「検証する」が偏りすぎないようにする。
6. 最大3件に絞る。
7. 参考カードだけになる場合は、強い断定表現を避ける。

### 7.10 表現ルール

- 「勝てる」「負ける」と断定しない。
- 「傾向がある」「次回確認したい」「勝ち筋を支えている可能性がある」のように、データから言える範囲に留める。
- 行動順やカード内容が必要な因果は書かない。
- 個人攻撃、能力断定、煽り表現を避ける。
- ネタ表現は見出しに限定してよいが、根拠行は堅く書く。

## 8. 開催回レビュー

### 8.1 対象開催回

初期対象は、選択中スコープ内で確定済み対戦が最も新しい開催回とする。

ユーザーは開催回セレクトで過去開催回へ切り替えられる。

### 8.2 ベースライン

開催回レビューの比較基準は、選択中の作品、シーズン、マップスコープとする。

選択スコープの件数が少なくレビュー根拠として弱い場合は、作品全体を補助ベースラインとして使ってよい。ただし画面上で「基準を作品全体に拡張」と明示する。

### 8.3 表示内容

開催回レビューは以下を表示する。

- 開催回名または開催日。
- 対象試合数。
- 第n試合ごとの勝者、資産差、物件収益トップ、スリの銀次合計、荒れ試合フラグ。
- 長期傾向と一致した点。
- 長期傾向とズレた点。
- 次回に持ち越す論点。

### 8.4 禁止事項

- 対象開催回だけを根拠に、長期的な課題として断定しない。
- 4戦未満または対象が少ない開催回で強い示唆を出さない。
- 全事件簿合算を総行動数、総移動量、総駅数として扱わない。

## 9. API / 実装方針

### 9.1 API単位

MVPでは、既存の選択肢APIと集計APIを維持しつつ、単一の振り返りAPIを追加する。

```text
GET /api/analytics/series-comparison/review?gameTitleId=...&seasonMasterId=...&mapMasterId=...&reviewHeldEventId=...
```

理由:

- 既存集計レスポンスを過度に肥大化させない。
- 開催回切り替え時に、詳細指標全体を再取得しなくてよい。
- 指標別APIを乱立させず、振り返りビュー単位の1 APIに閉じられる。

### 9.2 レスポンスの意味単位

振り返りレスポンスは、UIカード構造そのものではなく、以下の意味単位で返す。

- `baseline`: 比較基準スコープ、対象件数、補助ベースライン有無。
- `heldEventOptions`: レビュー対象にできる開催回候補。
- `selectedHeldEvent`: 現在レビュー対象の開催回。
- `eventReview`: 開催回レビューの要約、試合別点、長期傾向との差分。
- `playbookByPlayer`: プレーヤー別のプレイブックカード候補。
- `evidence`: カードごとの根拠指標、対象件数、信頼度、リンク先。
- `dataQuality`: サンプル不足、対象なし、同値あり、除外理由。

### 9.3 実装境界

- 推薦生成ロジックはAPI側に置くことを推奨する。
- webは表示用ViewModel、根拠リンク、空状態、エラー表示に集中する。
- 推薦生成の閾値、`priorWeight`、`actionability` はAPI側定数として集約し、テストで固定する。
- DBスキーマ変更はMVPでは不要とする。
- 対戦数増加で遅くなった場合だけ、キャッシュや集計テーブルを検討する。

## 10. 非機能要求

| 観点 | 要求 |
|---|---|
| 認証・権限 | 既存戦績比較ページと同じく、ログイン済みの有効アカウントだけが閲覧できる。 |
| セキュリティ | 参照専用APIとし、既存のCookie認証、dev/test account header本番無効化、Problem Details正規化に従う。 |
| データ正確性 | 順位は保存済み `rank` を正とし、総資産から再計算しない。 |
| 説明責任 | すべての提案に根拠指標、対象件数、信頼度、詳細リンクを付ける。 |
| 低コスト運用 | MVPでは外部推論エンジンを呼ばないため追加API課金は発生しない。 |
| 性能 | 固定4名、作品単位で数十〜百数十戦規模のデータを快適に表示できる。 |
| エラー表示 | レビューAPIだけが失敗しても、既存の戦績比較詳細タブは閲覧できる。 |
| アクセシビリティ | カード分類、信頼度、リンク先は色だけで意味を伝えない。 |
| レスポンシブ | PCでは4人のプレイブックを比較しやすく、モバイルでは1人ずつ読みやすい構成にする。 |

## 11. MVP範囲外

MVPでは以下を扱わない。

- AIによる自然文生成。
- Discord投稿文の自動生成。
- プレーヤー別テーマの保存。
- 次回開催後のテーマ達成判定。
- 複雑な予測モデル。
- カード内容、行動順、目的地距離など新規プレーデータの取得。
- 共有用画像生成。

将来AIを使う場合も、AIに推薦対象を決めさせない。決定論的に選ばれた根拠とカードを自然文へ整える用途に限定する。

## 12. テスト要求

実装時は、少なくとも以下を固定する。

- `view=review` と `reviewHeldEventId` のURL復元。
- 最新開催回の初期選択。
- 選択スコープ外の `reviewHeldEventId` の扱い。
- ベースライン件数不足時の補助ベースライン表示。
- 通常指標、条件付き指標の主採用、参考、非表示の境界。
- `rankScore`、効果量、縮小推定、`opportunityScore` の計算。
- プレーヤーごとの最大3件制限。
- 同じカテゴリから複数カードを出さないこと。
- 根拠チップの対象件数と信頼度表示。
- 既存タブへの根拠リンク。
- レビューAPI失敗時に既存集計表示を壊さないこと。

`docs/post-mortem/lessons.md` では、実装時に L1、L2、L3、L5、L7 が該当する。
