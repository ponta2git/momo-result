# 桃鉄の結果を記録し、集計するアプリケーションの要求仕様

毎週開催している深夜の桃鉄1年勝負の結果を記録し、後からCSV/TSVとして出力できるようにする。

本書は業務要求とMVP範囲の正典である。実装詳細は必要な範囲で記載するが、細かなライブラリ選定や運用手順は別途定める。

---

## 1. 概要

- 操作はWeb上で行う。
- 仲間内だけで運用する。
- ログイン可能な利用者は summit アプリの固定4名に限定する。
- 参加者も summit アプリの固定4名に限定する。
- 初回リリースのMVPでは、画像取り込み、OCR下書き作成、手修正、確定記録、CSV/TSV出力までを対象とする。
- 集計画面などの参照系UIは後から同じアプリケーション内に実装する。MVPでは参照画面よりもCSV/TSV出力を優先する。

---

## 2. summit アプリとの連携

`~/Documents/codes/summit` にある summit アプリと連携する。

- summit アプリのDBに相乗りする。
- summit アプリの `src/db/schema.ts` に共有スキーマを追記し、本アプリはそれを参照する。
- 将来的にスキーマ管理だけを別リポジトリまたは別パッケージに分離する可能性がある。
- summit アプリの以下の情報を共有する。
  - 固定4名のメンバー情報
  - 実開催履歴 `held_events`
- 本アプリ側でも開催履歴を作成できるが、開催履歴の正本は summit 共有の `held_events` に集約する。
- 1つの `held_events` には、第1試合、第2試合のように複数の桃鉄1年勝負の試合結果を紐づけられる。
- 試合番号は同じ `held_events` 内で自動採番し、必要に応じて手動変更できる。
- 結果作成時は開催履歴を選択し、その中に第n試合を追加する。開催履歴が存在しない場合は、本アプリから新規作成できる。

## 3. 認証・権限

### 3.1 ログイン

- ログイン方式は Discord OAuth とする。
- Discord OAuth のユーザーは summit の `members.user_id` と紐づける。
- 必要であれば、既存の `members.user_id` を不変の Discord ユーザーIDに移行する。
- ログイン可能なアカウントは固定4名だけに限定する。

### 3.2 管理者

- 管理者は固定4名のうち特定メンバーだけに付与する。
- 初期管理者は「ぽんた」とする。
- 管理者だけがログイン可能アカウントの許可を管理できる。

### 3.3 操作権限

- ログイン済みの固定4名は、誰でも以下を実行できる。
  - 開催履歴の作成
  - 試合結果の作成、修正、確定
  - 確定済み試合結果の修正
  - 確定済み試合結果の削除
  - 作品名、シーズン、事件名などのマスタ管理
  - CSV/TSVダウンロード
- 確定済み結果の修正履歴は保存しない。
- 開催履歴は、紐づく試合結果がない場合だけ固定4名が削除できる。

MVPでは以下の画面を提供する。

- ログイン画面
- 開催履歴一覧・作成画面
- 結果作成・編集画面
- 下書き一覧画面
- マスタ管理画面
- 管理者アカウント許可画面
- CSV/TSVダウンロード画面

## 5. 画像取り込み

### 5.1 取り込み方法

以下の両方に対応する。

- USB Video Class対応キャプチャーボードからのブラウザ撮影
- 画像ファイルアップロード

キャプチャーボード撮影UIには以下を含める。

- デバイス選択
- プレビュー
- 静止画撮影
- 再撮影
- 画像アップロードへの切替

対象ブラウザは Chrome / Firefox / Safari / Edge とする。ブラウザや端末の都合でキャプチャーボード撮影が使えない場合は、画像アップロードへフォールバックできればよい。

### 5.2 画像種別

取り込む画像の種類は以下の3種類である。

- 総資産
- 収益額
- 桃鉄事件簿

総資産と収益額の画像には、それぞれのプレーヤーと金額が順位順に表示されている。

桃鉄事件簿の画像には、各プレーヤーが遭遇したイベントの回数が表示されている。

画像の種類は自動判別する。ただし、誤判定時はユーザーが手動で種別を変更できる。

同じ種類の画像を複数取り込んだ場合は、最新の取り込み結果で上書きする。

### 5.3 画像の保存

- 画像は結果確定の必須条件ではない。
- 画像なしの手入力だけでも結果を確定できる。
- アップロード画像は1枚3MBまでとする。
- 取り込んだ画像はOCR完了までダウンロード可能にする。
- OCR完了後、画像はサーバーから削除する。
- 下書きには画像を残さない。
- サーバーに画像を恒久保存しない。

## 6. OCR・下書き

### 6.1 OCR処理

- OCR/画像解析には外部APIを使わず、ローカル処理に限定する。
- OCRはサーバー側で実行し、ブラウザは画像を一時アップロードする。
- OCR処理は、将来差し替え可能な構造にする。
- OCRジョブキューには Upstash Redis を使う。
- バックエンドはAPIサーバーとOCRワーカーを分離する。

### 6.2 下書き

- OCR結果は下書きとしてDBに保存する。
- 下書きはユーザーが修正・確認できる。
- 未確定下書きは管理者が削除するまでDB上に残す。
- OCRが失敗した場合は失敗理由を表示し、空の手入力下書きとして続行できる。
- OCR結果の信頼度が低そうな項目は、確認画面で強調表示する。

## 7. 結果記録

### 7.1 確定時の必須項目

試合結果の確定には以下を必須とする。

- 開催日時
- 作品名
- シーズン
- オーナー
- マップ
- 固定4名全員のプレー順
- 固定4名全員の順位

開催日時は `held_events` にあるものを自動入力する。ただし、手編集可能とする。

固定4名全員の順位は 1〜4 で重複なく入力されている必要がある。

### 7.2 プレイヤー別結果

プレイヤーごとに以下を記録する。

- プレー順
- 順位
- 総資産
- 収益額
- 事件簿の各項目の回数

総資産額と収益額は、万円単位の整数に正規化して保存する。

順位は総資産額から自動計算せず、OCRまたは手入力された順位を正として保存する。

プレー順は、画像内の色順から判別する。色順は青、赤、黄、緑の順で、画面下のインジケータ左から順番になっている。判別結果は手修正可能とする。

OCR上のプレーヤー名とDB内のメンバー名またはDiscord名が一致しない場合があるため、メンバーごとに複数の表示名エイリアスを持てるようにする。

### 7.3 オーナー・マップ

- オーナーは固定4名のいずれかとする。
- 作品にはマップ候補を紐づける。
- 試合結果ごとにオーナーとマップを入力する。

---

## 8. マスタ

MVPでは以下のマスタを扱う。

- 作品
- マップ
- シーズン
- 事件名
- プレーヤー名エイリアス

### 8.1 作品・マップ

- 作品はあらかじめマスタとして登録する。
- マップは作品に紐づける。

### 8.2 シーズン

- シーズンは期間で区切る大会・集計単位である。
- 1つのシーズンは複数の開催結果を束ねる。
- 結果確定時はシーズン選択を必須とする。

### 8.3 事件名

桃鉄事件簿は、事件名マスタとプレイヤーごとの回数として構造化して記録する。

MVPでは以下の6項目を固定で扱う。

- 目的地
- プラス駅
- マイナス駅
- カード駅
- カード売り場
- スリの銀次

---

## 9. CSV/TSV出力

### 9.1 権限

- CSV/TSVダウンロードは非公開APIとする。
- ログイン中UIからのみ実行できる。
- ログイン済み固定4名全員がダウンロードできる。

### 9.2 出力範囲

以下の範囲を指定してCSV/TSVを出力できる。

- 全試合
- シーズン単位
- `held_events` 単位
- 試合単位

### 9.3 行構造

CSV/TSVは、1行を1プレイヤーの試合結果明細とする。

列は以下の順序を基本とする。

| 列 | 内容 |
|---|---|
| シーズン | シーズン名 |
| シーズンNo. | シーズン内の対戦連番 |
| オーナー | 試合結果ごとに入力されたオーナー |
| マップ | 試合結果ごとに入力されたマップ |
| 対戦日 | 開催日 |
| 対戦No. | 同じ作品内の通算対戦番号 |
| プレー順 | プレイヤーのプレー順 |
| プレーヤー名 | プレイヤー名 |
| 順位 | 最終順位 |
| 総資産 | 万円単位の総資産 |
| 収益 | 万円単位の収益額 |
| 目的地 | 桃鉄事件簿の回数 |
| プラス駅 | 桃鉄事件簿の回数 |
| マイナス駅 | 桃鉄事件簿の回数 |
| カード駅 | 桃鉄事件簿の回数 |
| カード売り場 | 桃鉄事件簿の回数 |
| スリの銀次 | 桃鉄事件簿の回数 |

---

## 10. 技術・構成

- フロントエンドは React/Vite のSPAとする。
- バックエンドAPIとフロントエンドは分離する。
- バックエンドは APIサーバー + OCRワーカーを同一リポジトリ、同一Fly.io構成内で分離する。
- バックエンドは Fly.io を使う。
- DBは Neon を使い、summit アプリと共有する。
- OCRジョブキューには Upstash Redis を使う。
