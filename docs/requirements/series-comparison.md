# 戦績比較ページ 要求仕様（シリーズ別）

本書は、確定済みの戦績データを使って、選択したシリーズ内の対戦結果をプレーヤー間で比較する参照ページの要求仕様である。

`docs/requirements/base.md` のMVP後に追加する集計画面として扱う。既存の試合確定、OCR、出力系仕様は変更しない。

---

## 0. 最終判断

初期実装は、シリーズ同士を横断比較するページではなく、1つのシリーズを選んで固定4名を比較する「対戦後の成績ボード」として作る。

最終的な推奨ライン:

- 「シリーズ」は既存の「作品」(`game_titles`) と同義として扱う。
- 初回表示は、確定済み対戦が最も新しいシリーズの「総合」を自動選択する。今後も集計対象は確定済み試合だけに限定する。
- APIは単一巨大API、完全な指標別API、2系統APIを比較したうえで、初期版では「選択肢取得API + 集計API」の2系統を推奨する。
- 画面は指標ブロックを縦に並べ、縦スクロールで読む構成にする。PCでは各指標ブロック内で4プレーヤーを横並び比較できるようにする。
- 指標は `metricId` 付きのカタログとして定義し、説明、数式、良い方向、サンプル不足条件を持たせる。
- ネタ要素は入れるが、根拠指標と対象件数を必ず表示する。
- 事件簿の合算を総行動数として扱わない。
- 参照系ページだが公開画面にはしない。既存のログイン後ページと同じ認証範囲内に置く。
- 初期版ではDBスキーマ変更をしない。専用読み取りAPIで都度集計する。

## 1. 目的と成功条件

### 1.1 目的

- 固定4名の桃鉄1年勝負の戦績を、シリーズ単位で比較できるようにする。
- 各指標を縦に並べて読み進めながら、指標ごとに4人の差が分かる画面にする。
- 単なる成績表ではなく、勝ち筋、事故耐性、収益と順位のズレなど、対戦後に会話が盛り上がる材料を出す。
- 複雑な指標は、見出し、数式、解釈をページ内で確認できるようにする。
- 事件簿の各項目を合算して「総行動数」のように扱わない。事件簿に含まれない駅・行動があるため、合算値は行動総量ではない。

### 1.2 成功条件

- 利用者がシリーズを選ぶだけで、対象対戦数と4人の主要成績差を30秒以内に把握できる。
- URLを直接開いた場合でも、無効なqueryでない限り同じスコープが復元される。
- PC幅では、縦に並んだ各指標ブロックの中で、各プレーヤーを横並びにして同一指標を比較できる。
- 平均順位、入賞率、銀次、収益外成績、目的地関連のどれかを根拠に、対戦後の話題を作れる。
- 複雑な指標は、画面上の説明だけで「何を測っているか」「高い/低いとどう読むか」が分かる。
- データが少ない、対象がない、同値が多い場合に、誤解を招く断定表示をしない。

## 2. 用語とデータ前提

| 用語 | 意味 |
|---|---|
| シリーズ | 初期定義では `game_titles` の1件、つまり既存UIの「作品」と同義とする。 |
| 対戦 | `matches` の確定済み試合1件。未確定下書きは含めない。 |
| プレーヤー | `members` の固定4名。 |
| シーズン | `season_masters`。シリーズに紐づく集計単位。 |
| マップ | `map_masters`。シリーズに紐づく集計単位。 |
| 対戦順 | `played_at`、`held_event_id`、`match_no_in_event`、`match_id` の昇順で並べたシリーズ内通算順。既存の対戦番号付けと同じ考え方にする。 |
| 指標カタログ | 指標ID、表示名、説明、数式、単位、良い方向、サンプル不足判定を持つ定義一覧。 |

データ前提:

- 集計対象は確定済み `matches` のみとする。
- `match_players` の `rank`、`play_order`、`total_assets_man_yen`、`revenue_man_yen` を使う。
- `match_incidents` の固定6項目を使う。
- 金額はDB保存値どおり万円単位の整数として集計する。
- 順位はDB保存値を正とし、総資産から再計算しない。
- 既存仕様では固定4名全員が全試合に参加する。将来変更に備え、指標の分母はプレーヤーごとの参加対戦数として定義する。

## 3. 対象範囲

### 3.1 初期版で対象にすること

- シリーズ選択。
- 対戦回数表示。
- 総合、シーズン別、マップ別の切り替え。
- 指標ブロックを縦に並べた縦スクロール画面。
- PCでは各指標ブロック内で4プレーヤーを横並び比較できる画面。
- 基本成績、入賞率、順番補正、スリの銀次、収益外成績、目的地関連の集計。
- 推移グラフとヒストグラム。
- 指標説明。
- ネタ・戦略的ハイライト。
- 認証済みユーザー間でのURL query による選択状態の共有。

### 3.2 初期版で対象外にすること

- 試合結果の作成・編集。
- OCR下書きや未確定下書きの集計。
- 事件簿合算による総行動数、総駅数、総マス数の推定。
- マスタ管理。
- CSV/TSV出力。
- 共有用画像生成。
- AI連係またはAIによる自由文生成。
- シリーズ同士の同時比較。初期版は1シリーズ内のプレーヤー比較に集中する。

### 3.3 段階分け

| 段階 | 内容 |
|---|---|
| Phase 1 | 1シリーズ内の総合、シーズン別、マップ別比較。主要指標、推移、ヒストグラム、ハイライト。 |
| Phase 2 | シーズンとマップの同時絞り込み、直近n戦の移動平均、オーナー別、プレー順別、開催回別、シリーズ間比較。 |
| Phase 3 | 任意指標のON/OFF、詳細な事件簿分布、複数スコープを並べる比較表示。 |

Phase 2以降は、本書の拡張性方針に従って追加する。ただし、共有用画像生成、試合結果の作成・編集、マスタ管理、CSV/TSV出力、AI連係は本ページの拡張候補に含めない。

## 5. IA要求

### 5.1 ナビゲーション

- 通常ページとしてグローバルナビに追加する。
- ナビゲーション名は「戦績比較」を推奨する。既存の「試合」「OCR」「開催」「出力」と並べても意味が通りやすい。
- ルート候補は `/analytics/series-comparison` とする。実装時に既存ルーティング規約と合わせて最終決定する。
- 実装時は既存のログイン後ページと同じく `AuthenticatedRoute` 配下に置き、`AppShell` 内で表示する。
- 未ログイン時はログイン画面へ戻り先付きで遷移し、利用不可アカウントは既存の権限エラー表示に従う。

### 5.2 ページ構造

ページは以下の順に構成する。

1. コントロールバー: シリーズ選択、スコープ切り替え、対象対戦数。
2. 主要サマリー: 重要な数値とハイライトを短く表示する。
3. 指標ブロック一覧: 平均順位、順位分布、総資産、収益、入賞・沈没、順番補正、銀次、収益外成績、目的地関連を縦に並べる。
4. 指標説明: 各指標ブロック内の説明ポップオーバーまたは補助パネル。

画面は縦スクロールを前提にする。指標ブロック同士を横に敷き詰めるダッシュボード型にはしない。

初回表示:

- URL query が有効なら、その状態を復元する。
- URL query がない場合は、確定済み対戦が最も新しいシリーズの `overall` を選ぶ。
- 確定済み対戦がないがシリーズマスタだけ存在する場合は、先頭表示順のシリーズを選び、対戦なしの空状態を出す。
- URL query は表示状態の復元だけに使い、未ログイン利用者への公開共有手段にはしない。

### 5.3 スコープモデル

- `gameTitleId` は必須スコープとする。
- `scopeKind` は `overall`、`season`、`map` のいずれかとする。
- `scopeId` は `scopeKind=season` のとき `seasonMasterId`、`scopeKind=map` のとき `mapMasterId` を指す。
- 初期版ではシーズンとマップの同時絞り込みはしない。
- URL query の候補は `gameTitleId`、`scopeKind`、`scopeId` とする。
- `scopeKind=overall` のとき `scopeId` は持たない。URLに付いている場合は無効queryとして扱う。
- 無効な `scopeId` は選択シリーズ内の初期スコープに戻し、利用者に通常の空状態または軽い通知で知らせる。

### 5.4 指標ブロック

- 1つの指標ブロックは、見出し、短い読み方、4人比較、必要ならグラフまたは分布で構成する。
- ブロックはページ上で縦に並べる。
- ブロック見出しには、必要に応じて説明アイコンと信頼性ラベルを置く。
- ブロック内の比較テーブルでは、行に指標内のサブ項目、列にプレーヤーを置くのを基本とする。
- グラフがあるブロックでは、グラフの直下または横に4人の数値比較を置き、グラフだけで読ませない。

### 5.5 PCレイアウト

- 4人のプレーヤー列を各指標ブロック内の比較軸にする。
- スクロールでプレーヤー名が見失われないよう、セクション見出し直下にプレーヤーヘッダーを繰り返すか、横スクロール領域内で固定する。
- 数値の単位、順位の良し悪し、平均との差は列間で揃える。
- グラフは、1つのグラフに全プレーヤーを重ねるものと、プレーヤー別分布を並べるものを使い分ける。

### 5.6 モバイルレイアウト

- 4人横並びは必須にしない。
- プレーヤー切り替え、横スクロール、または2列カードで破綻しない表示にする。
- グラフは横幅優先で表示し、詳細表は横スクロールを許容する。
- PCとモバイルで表示密度は変えてよいが、指標定義、対象スコープ、分母は同一にする。

## 6. UX要求

### 6.1 利用者と利用シーン

- 利用者はいつもの対戦メンバーと運用者。
- 利用シーンは、対戦後の振り返り、Discordでの雑談、過去シリーズの振り返り。
- 精密な分析ツールよりも、数字を根拠に会話できる「対戦後の成績ボード」を目指す。

### 6.2 見た目の方向性

ドメイン:

- 駅、目的地、マップ、順位、総資産、収益、銀次、事件簿、対戦順。

色の世界:

- 盤面、駅看板、路線、順位表、カード、目的地マーカー、銀次の注意色。

シグネチャ:

- 「縦に続く成績巻物」と「4人の社長列」を組み合わせる。ページ全体は縦に読み、各指標ブロック内ではPCで4人を横方向に比較できる構造にする。

避ける既定パターン:

- 汎用KPIカードを大量に並べるだけのダッシュボード。
- 1人ずつ縦に積むだけで比較しづらいプロフィール画面。
- 指標ブロックを横に詰め込み、スクロール方向や比較軸がばらばらになる画面。
- 指標名だけを出して数式や解釈を隠す分析画面。

### 6.3 説明と納得感

- 複雑な指標には、見出し横に説明アイコンを置く。
- 説明には、何を測るか、数式、値が高い/低い場合の読み方、サンプル数が少ない場合の注意を含める。
- 説明が必要な指標は、順番補正、銀次耐性、収益外順位、高収益未勝利率、目的地変換効率、目的地依存度とする。
- 「参考」「対象なし」「同値あり」などの信頼性ラベルは、見た目だけでなくテキストで表示する。

### 6.4 グラフ表示

- 順位系グラフは、1位が上に見えるようにY軸を反転する。
- 色だけで系列を区別しない。凡例、線種、点、ラベルを併用する。
- ヒストグラムは全プレーヤーで同じbin境界を使う。
- サンプル数が少ないスコープでは、ヒストグラムを強く読ませすぎないよう「参考」表示を付ける。
- グラフには、キーボード操作とスクリーンリーダーで意味が取れるラベルを付ける。

### 6.5 ハイライト表示

- ネタ表示は、主要指標の信頼性を壊さないよう、詳細指標の後またはサマリー下部に置く。
- ハイライト名は桃鉄文脈で短く遊びを入れてよいが、個人攻撃に見える言い回しは避ける。
- ハイライトの根拠行は堅く書き、対象試合数、根拠指標、同値の有無を必ず表示する。
- 対象試合が少ないハイライトは出さないか、「参考」と明記する。

## 7. 指標定義

### 7.1 共通ルール

- 各プレーヤーの分母は、そのプレーヤーが選択スコープ内で参加した確定済み対戦数とする。
- 確率は `該当回数 / 分母` とする。
- 平均は算術平均とする。
- 中央値は昇順に並べ、件数が偶数の場合は中央2値の平均とする。
- 金額の最高、最低、平均、中央値は万円単位のDB値から計算し、UI表示で億円などに整形してよい。
- 値順位を計算する場合、最終順位以外は同値が起こり得る。同値順位は平均順位方式を基本とする。
- 「トップだったとき」の判定は、その試合の最大値と等しい全員をトップ扱いにする。
- 平均順位方式を使った指標では、同値が1件以上あればAPIが `hasTies = true` 相当のメタ情報を返し、UIは必要に応じて「同値は平均順位で計算」と表示できる。
- サンプル数が3未満の分母を持つ指標は「参考」と表示する。
- 分母が0の指標は「対象なし」と表示する。
- 小数は表示上丸めてよいが、計算途中では丸めない。
- 標準偏差は、選択スコープ内の対象データ全体を母集団として扱う母標準偏差とする。

### 7.2 対戦回数

- 選択スコープ内の確定済み対戦数を表示する。
- シリーズを選択していない状態では表示しない。

### 7.3 基本情報

各プレーヤーごとに以下を表示する。

| 指標ID | 指標 | 定義 |
|---|---|---|
| `rank.average` | 平均順位 | `rank` の平均。小さいほど良い。 |
| `rank.cumulativeAverageTrend` | 平均順位の遷移グラフ | 対戦順に沿った累積平均順位。1つの折れ線グラフに全プレーヤーを重ねる。 |
| `rank.distribution` | 各順位の回数・確率 | 1位、2位、3位、4位それぞれの回数と確率。 |
| `assets.max` | 総資産 最高額 | `total_assets_man_yen` の最大値。 |
| `assets.min` | 総資産 最低額 | `total_assets_man_yen` の最小値。 |
| `assets.average` | 総資産 平均値 | `total_assets_man_yen` の平均。 |
| `assets.median` | 総資産 中央値 | `total_assets_man_yen` の中央値。 |
| `assets.histogram` | 総資産ヒストグラム | 選択スコープ内の全プレーヤー値を基準に共通binを作り、プレーヤー別分布を比較する。初期bin数は最大8を目安にAPI側で返す。 |
| `revenue.max` | 収益 最高額 | `revenue_man_yen` の最大値。 |
| `revenue.average` | 収益 平均値 | `revenue_man_yen` の平均。 |
| `revenue.median` | 収益 中央値 | `revenue_man_yen` の中央値。 |
| `revenue.histogram` | 収益ヒストグラム | 選択スコープ内の全プレーヤー値を基準に共通binを作り、プレーヤー別分布を比較する。初期bin数は最大8を目安にAPI側で返す。 |

収益の最低値は初期表示に含めない。収益は借金・事故の影響で極端な値が出やすく、会話上の主指標として扱いにくいためである。

### 7.4 入賞・沈没

見出し候補は「入賞・沈没」とする。

| 指標ID | 指標 | 定義 |
|---|---|---|
| `podium.count` | 入賞回数 | `rank IN (1, 2)` の回数。 |
| `podium.rate` | 入賞率 | 入賞回数 / 分母。 |
| `podium.cumulativeRateTrend` | 入賞率推移グラフ | 対戦順に沿った累積入賞率。1つの折れ線グラフに全プレーヤーを重ねる。 |
| `lowerHalf.count` | 沈没回数 | `rank IN (3, 4)` の回数。 |
| `lowerHalf.rate` | 沈没率 | 沈没回数 / 分母。 |
| `lowerHalf.cumulativeRateTrend` | 沈没率推移グラフ | 対戦順に沿った累積沈没率。1つの折れ線グラフに全プレーヤーを重ねる。 |

### 7.5 順番補正

見出し候補は「順番補正」とする。

プレー順による有利不利を見るため、以下を表示する。

定義:

- `playOrderBaseline(metric, order)` は、選択スコープ内で `play_order = order` の全プレーヤー行における対象指標の平均とする。
- `playerAdjustedDiff(metric)` は、対象プレーヤーの各対戦行で `value - playOrderBaseline(metric, play_order)` を計算し、その平均を取る。
- `playerAdjustedIndex(metric)` は、対象プレーヤーの各対戦行で `value / playOrderBaseline(metric, play_order)` を計算し、その平均を取る。
- baseline には対象プレーヤー自身の行も含める。leave-one-out補正は初期版では採用しない。

| 指標ID | 指標 | 定義 |
|---|---|---|
| `playOrder.assetsDiff` | 総資産 順番補正差 | `playerAdjustedDiff(total_assets_man_yen)`。 |
| `playOrder.revenueDiff` | 収益 順番補正差 | `playerAdjustedDiff(revenue_man_yen)`。 |
| `playOrder.assetsIndex` | 総資産 順番補正指数 | `playerAdjustedIndex(total_assets_man_yen)`。 |
| `playOrder.revenueIndex` | 収益 順番補正指数 | `playerAdjustedIndex(revenue_man_yen)`。 |

推奨主表示は「差」とする。金額は負値になり得るため、比率は分母が0付近または負値のとき解釈が壊れやすい。指数は補助表示とし、分母が0以下または絶対値が小さい場合は表示しない。

### 7.6 スリの銀次

見出し候補は「銀次まわり」とする。

| 指標ID | 指標 | 定義 |
|---|---|---|
| `ginji.count` | 遭遇回数 | `incidents.suriNoGinji` の合計。 |
| `ginji.cumulativeCountTrend` | 遭遇回数グラフ | 対戦順に沿った累積遭遇回数。1つの折れ線グラフに全プレーヤーを重ねる。 |
| `ginji.encounterRate` | 遭遇率 | `incidents.suriNoGinji >= 1` の対戦数 / 分母。 |
| `ginji.resilienceRankAverage` | 銀次耐性 平均順位 | `incidents.suriNoGinji >= 1` の対戦だけを対象にした平均順位。 |
| `ginji.resilienceAssetsAverage` | 銀次耐性 平均総資産 | `incidents.suriNoGinji >= 1` の対戦だけを対象にした平均総資産。 |
| `ginji.resilienceRevenueAverage` | 銀次耐性 平均収益 | `incidents.suriNoGinji >= 1` の対戦だけを対象にした平均収益。 |

銀次耐性の対象試合が0件の場合は「対象なし」と表示する。

### 7.7 収益外成績

見出し候補は「稼ぎ以外の勝ち筋」とする。

| 指標ID | 指標 | 定義 |
|---|---|---|
| `nonRevenue.rankDelta` | 収益外順位 | `平均収益順位 - 平均総合順位`。正の値が大きいほど、収益順位以上の最終順位を取っている。 |
| `nonRevenue.highRevenueNoWinCount` | 高収益未勝利回数 | その試合で収益トップだったが、最終順位が1位ではなかった回数。 |
| `nonRevenue.highRevenueNoWinRate` | 高収益未勝利率 | 高収益未勝利回数 / 収益トップ回数。 |

収益順位は、各試合の `revenue_man_yen` 降順で計算する。収益トップ回数が0の場合、高収益未勝利率は「対象なし」と表示する。

`nonRevenue.rankDelta` は、正なら「収益順位より総合順位が良い」、負なら「収益順位より総合順位が悪い」と読む。

### 7.8 目的地関連

見出し候補は「目的地との付き合い方」とする。

初期版では、解釈しやすい指標を2つ表示する。

| 指標ID | 指標 | 定義 |
|---|---|---|
| `destination.conversionDelta` | 目的地変換効率 | `平均目的地順位 - 平均総合順位`。正の値が大きいほど、目的地到着順位以上の最終順位を取っている。 |
| `destination.dependenceScore` | 目的地依存度 | `目的地上位時の平均順位スコア - 目的地下位時の平均順位スコア`。正の値が大きいほど、目的地を取れた試合で成績が伸びている。 |

目的地順位は、各試合の `incidents.destination` 降順で計算する。

順位スコアは `5 - rank` とする。1位=4点、2位=3点、3位=2点、4位=1点。

目的地依存度の分類:

- 目的地上位は、その試合内の目的地順位が平均順位方式で2.5未満の行とする。
- 目的地下位は、その試合内の目的地順位が平均順位方式で2.5より大きい行とする。
- 目的地順位が2.5ちょうどの行は中立として依存度計算から除外する。
- 上位または下位の対象試合が0件の場合は「対象なし」と表示する。

### 7.9 ネタ・戦略的ハイライト

初期版で入れる候補を以下とする。すべて選択スコープ内の集計結果から決定論的に算出する。

| ハイライトID | ハイライト | 内容 |
|---|---|---|
| `highlight.ginjiResilience` | 銀次リカバリー王 | 銀次遭遇試合の平均順位が最も良いプレーヤー。対象試合が少ない場合は出さない。 |
| `highlight.highRevenueNoWin` | 稼ぎ空振り注意報 | 高収益未勝利率が最も高いプレーヤー。 |
| `highlight.destinationCraft` | 目的地職人 | 目的地依存度が最も高いプレーヤー。 |
| `highlight.destinationIndependent` | 寄り道勝ち筋 | 目的地変換効率が最も高いプレーヤー。 |
| `highlight.assetsPeak` | 資産ピーク王 | 総資産最高額が最も高いプレーヤー。 |
| `highlight.revenuePeak` | 収益爆発王 | 収益最高額が最も高いプレーヤー。 |
| `highlight.stability` | 安定社長 | 順位の標準偏差が最も小さいプレーヤー。 |

## 8. データ取得・API要求

### 8.1 方針

- 既存の試合一覧APIは詳細な金額・事件簿を返さないため、初期実装では専用の読み取りAPIを追加する。
- フロントエンドが試合一覧から各試合詳細を大量取得して集計する実装は避ける。
- DBスキーマ変更は初期版では不要とする。
- 集計ロジックはAPI側に置く。webは表示用ViewModelへの変換に留める。

### 8.2 API単位の検討

| 案 | 内容 | メリット | デメリット | 判定 |
|---|---|---|---|---|
| 単一API | 選択肢、現在スコープの集計、指標定義を1つのAPIで返す。 | 初回表示のHTTPリクエストが1回で済む。同時接続数を最小化できる。 | スコープ変更のたびに変化しにくい選択肢まで再取得する。指標追加時にレスポンスが肥大化しやすく、cache粒度が粗い。 | 不採用 |
| 2系統API | 選択肢取得APIと集計APIに分ける。 | 選択肢は長めにcacheでき、集計はスコープ単位でcacheできる。初回でも最大2リクエストで済み、同時接続数を抑えられる。指標追加時も集計DTOに閉じやすい。 | 初回表示で選択肢取得後に集計取得が必要。 | 採用 |
| 指標別API | 指標ブロックごとにAPIを分ける。 | 指標追加や遅延読み込みはしやすい。重い指標だけ後から取得できる。 | 縦に並ぶ指標数だけリクエストが増え、同時接続数、ローディング状態、失敗表示、整合性管理が複雑になる。 | 初期版では不採用 |

初期版では2系統APIを採用する。縦スクロール画面で多数の指標ブロックを出すが、指標別APIにはしない。指標追加時のメンテナンス性は、API分割ではなく `metricId` と指標カタログ、DTOの意味単位分割で担保する。

同時接続数への配慮:

- 初回は選択肢APIを取得し、既定スコープを確定してから集計APIを取得する。
- URL query が有効で、選択肢がcache済みの場合は、集計APIをすぐ取得してよい。
- 指標ブロックごとの並列fetchは行わない。

### 8.3 選択肢取得API

ページ初期表示とセレクト候補のため、集計とは別に軽量な選択肢APIを用意する。

```text
GET /api/analytics/series-comparison/options
```

レスポンスには以下を含める。

- シリーズ候補。
- シリーズごとのシーズン候補。
- シリーズごとのマップ候補。
- 確定済み対戦が最も新しいシリーズID。
- 各シリーズの確定済み対戦数。

候補APIは、集計値やグラフ点を返さない。
認証済みの有効アカウントだけが取得できる。

### 8.4 集計API

推奨エンドポイント:

```text
GET /api/analytics/series-comparison?gameTitleId=...&scopeKind=overall|season|map&scopeId=...
```

- `gameTitleId` は必須。
- `scopeKind` は省略時 `overall` とする。
- `scopeId` は `scopeKind=season` または `scopeKind=map` のとき必須とする。
- `scopeKind=overall` で `scopeId` が指定された場合は `400` とする。
- 認証済みの有効アカウントだけが取得できる。
- 無効なシリーズ、スコープ、シリーズ配下ではないスコープIDは `400` または `404` の Problem Details として返す。

レスポンスには以下を含める。

- `schemaVersion`。
- 選択スコープ情報。
- 選択中スコープの表示名。
- 対戦回数。
- プレーヤー一覧と表示順。
- 各指標の集計値。
- 推移グラフ用の時系列データ。
- ヒストグラム用のbin境界とbin別件数。
- 指標ごとの分母、対象件数、サンプル不足フラグ。
- 同値計算のメタ情報。
- ハイライトと根拠指標。

### 8.5 API DTOの考え方

レスポンスは、UIコンポーネント都合のカード構造ではなく、以下の安定した意味単位で返す。

- `scope`: 集計範囲。
- `players`: プレーヤー表示情報。
- `metricsByPlayer`: プレーヤー別の指標値。
- `trends`: 時系列グラフ。
- `histograms`: 分布グラフ。
- `highlights`: ハイライト。
- `dataQuality`: サンプル不足、対象なし、同値あり、除外行数などの表示補助。
- `metricDefinitions`: 画面説明に必要な指標カタログ。

`metricDefinitions` はAPIが返すか、web側の同一バージョン定数として持つ。どちらにする場合も、`metricId` と説明内容がテストで固定される必要がある。

新しい指標を追加する場合は、原則として集計APIの `metricsByPlayer`、`trends`、`histograms`、`highlights`、`metricDefinitions` のいずれかへ意味単位で追加する。スコープ候補が増えない限り、選択肢取得APIは変更しない。

### 8.6 性能

- 固定4名運用を前提に、1シリーズ全件を1リクエストで取得して快適に表示できること。
- 集計SQLは `matches`、`match_players`、`match_incidents`、マスタを結合するため、DB-backed repository test の対象にする。
- 将来データ量が増えた場合でも、ページ表示のためにN+1で試合詳細APIを呼ばない。
- 初期版では都度集計でよい。対戦数が増えて体感遅延が問題になった場合だけ、materialized view や集計テーブルを検討する。

| キャッシュ | TanStack Query の query key は、選択肢APIと集計APIで分ける。集計APIの key は `gameTitleId`、`scopeKind`、`scopeId`、response shape version を含める。raw DTO とViewModelを同じkeyに置かない。 |
| エラー表示 | `query.error` / `isError` だけで致命的失敗扱いにしない。認証、`enabled`、`isFetching`、cached error、refetch success を考慮する。 |

| レスポンシブ | PCでは4人横並び比較を必須にする。スマホでは破綻しない代替表示を用意する。 |
| ブラウザ | 既存web要件と同じく、Chrome / Firefox / Safari / Edge の最新安定版を対象にする。 |
| 観測性 | API失敗時は request id、account id、scope、集計対象件数、error code を構造化ログに残す。レスポンス全体やセッション情報はログに出さない。 |
| 低コスト運用 | Fly.io単一VM構成を前提に、重いバックグラウンド集計や常時更新処理を初期版で導入しない。 |

## 10. 拡張性要求

### 10.1 指標追加

- 指標は `metricId` を持つカタログとして追加する。
- 指標カタログには、表示名、カテゴリ、説明、数式、単位、良い方向、サンプル不足条件、表示可否条件を持たせる。
- 新しい指標を追加するときは、既存指標IDの意味を変えない。
- 指標の意味を変える場合は、新しい `metricId` または `schemaVersion` を使う。

### 10.2 スコープ追加

将来追加しやすいスコープ候補:

- オーナー別。
- プレー順別。
- 開催回別。
- 直近n戦。
- シリーズ間比較。
- 任意期間。

初期版の `scopeKind` / `scopeId` モデルは、これらを追加できる形で実装する。

### 10.3 事件簿拡張

- 事件簿マスタが将来増えても、固定6項目前提の指標と、事件マスタ一覧を使った汎用分布表示を分ける。
- 「総行動数」と誤解される合算指標は追加しない。
- 新しい事件項目をハイライト対象にする場合も、事件簿に含まれない行動がある注意を維持する。

### 10.4 集計方式の拡張

- 初期版は都度集計で実装する。
- 対戦数増加で遅くなった場合は、API内部でキャッシュ、materialized view、集計テーブルを検討する。
- materialized view や集計テーブルを導入する場合は、`../momo-db` のmigration、DB contract test、更新タイミング、古い集計の表示ポリシーを別途仕様化する。

### 10.5 表示拡張

- グラフコンポーネントは、平均順位、率、累積回数、ヒストグラムを別々の入力型で扱う。
- 指標追加でUI全体が崩れないよう、カテゴリごとに表示順を持つ。
- 将来の画面内表示拡張では、画面表示用の装飾ではなく、集計DTOと指標カタログを再利用する。
- 共有用画像生成、CSV/TSV出力、AI連係は本ページの表示拡張候補に含めない。

## 11. 空・エラー・境界ケース

- シリーズが未登録の場合、シリーズ選択に空状態を出す。
- 選択シリーズに確定済み対戦がない場合、集計表ではなく空状態を出す。
- 特定のシーズンまたはマップに対戦がない場合、そのスコープを選ぶと空状態を出す。
- 分母が0の指標は「対象なし」と表示する。
- 分母が3未満の指標は「参考」と表示する。
- 金額が負値の場合も集計対象に含める。
- ヒストグラムのbinは、負値と正値が混在しても全プレーヤーで同じ境界を使う。
- 事件簿項目が欠けた古いデータがある場合は0として扱う。ただしDB契約上は現行データでは6項目が保存される想定。
- 収益トップや目的地順位で同値がある場合は、共通ルールの同値処理に従う。
- 選択中にマスタが削除または更新された場合は、APIのエラーまたは候補再取得に従って有効なスコープへ戻す。
- URL query が不正な場合は、初期スコープに戻す。復元できない値を黙って別スコープとして解釈しない。
