# 戦績比較 振り返り・行動プレイブック要求仕様

本書は、`docs/requirements/series-comparison.md` で定義済みの戦績比較ページに追加する「振り返り」機能の要求仕様である。

既存の試合確定、OCR、CSV/TSV出力、マスタ管理、戦績比較の基礎指標は変更しない。確定済み戦績データだけを使い、プレーヤーに対して次回4戦で試す「試合中の行動仮説」を提示する。

---

## 0. 最終判断

MVPの推奨ライン:

- AI連係、AIによる自由文生成、外部推論エンジンはMVPに含めない。
- 既存の戦績比較ページ内に、分析サブページタブとして「振り返り」を追加する。別ページ化しない。
- 「振り返り」は分析タブの先頭に置く。ただし既存の詳細指標タブは根拠の正本として維持する。
- 「振り返り」は期間プレイブックに集約する。開催回レビュー、開催回選択、開催回単位の要約はMVPに含めない。
- 期間プレイブックは、選択中の作品、シーズン、マップの集計から、各プレーヤーに最大3件の「試合中に試す行動仮説」を出す。
- 既存指標の詳細表やグラフは振り返りタブで再掲しない。一方で、行動仮説を理解するための症状、差分、主要な数値はカード内に表示し、詳細リンクだけに依存しない。
- 上部サマリーや既存 `highlights` と重複する話題性のある結論は、振り返りタブへ寄せる。既存タブ側は詳細根拠を読む場所にする。
- 推薦は決定論的に生成する。順位スコア、症状スコア、差分スコア、対象件数、縮小推定、信頼度、行動接続度を使い、サンプル不足による過剰な断定を避ける。
- 新しいプレーデータは取得しない。行動順、カード内容、目的地距離、移動距離、リアルタイム意思決定は推定しない。

## 1. 目的と提供価値

### 1.1 目的

既存の戦績比較ページは、平均順位、資産分布、収益、目的地、番手、スリの銀次などを詳細に比較できる。一方で、対戦後の振り返りでは、利用者が複数タブの数値を読んで「結局、次の試合中にどの場面で何を優先し、何を避けるか」へ変換する必要がある。

振り返り機能の目的は、既存指標を増やすことではなく、蓄積済み戦績から以下を短時間で得られるようにすることである。

- 各プレーヤーが次回の4戦で試すべき行動仮説は何か。
- その行動仮説は、どんな場面で発動するのか。
- 試合中に何を優先し、何を避けるべきか。
- その仮説の根拠となる症状、差分、主要数値は何か。
- 既存の詳細指標へ移動すれば、カード内の読み取りをさらに確認できること。

### 1.2 提供価値

| 観点 | 内容 |
|---|---|
| 背景 | 既存の詳細分析は有用だが、振り返り時に「次回へ持ち越す論点」へ読み替える負担がある。 |
| 何を行うか | 選択スコープ全体の傾向から、各プレーヤーに最大3件の試合中の行動仮説を提示する。 |
| どのくらい | 20〜30分かかり得る雑談的な読み解きを、5分程度で論点化する。各自の行動仮説は1〜3件に絞る。 |
| どんな価値 | 勝ち筋の再現、崩れ条件の把握、次回4戦で試す行動方針の共有を支援する。 |

### 1.3 成功条件

- 利用者が振り返りタブを開いてから30秒以内に、各プレーヤーの次回4戦で試す行動仮説を把握できる。
- 各プレーヤーに0〜3件のプレイブックカードが表示され、カードごとに「再現する」「見直す」「検証する」のいずれかが分かる。
- すべてのカードに、発動条件、やること、避けること、データ上の理由、試合後の検証方法、対象件数、信頼度、詳細タブへのリンクがある。
- カード本文だけで、次回の試合中に何を試すかが分かる。
- 対象件数が少ない条件を、強い行動仮説として断定しない。
- 振り返りタブ内で既存指標の表やグラフを重複表示しない。
- 既存タブへ移動すれば、カードの根拠となる詳細指標を確認できる。

## 2. 用語

| 用語 | 意味 |
|---|---|
| 振り返り | 戦績比較ページに追加する分析サブページ。期間プレイブックを表示する。 |
| 期間プレイブック | 選択中の作品、シーズン、マップの確定済み戦績から作る、プレーヤー別の行動仮説セット。 |
| 行動仮説 | 次回4戦の試合中に試す具体的な方針。最適手の断定ではなく、戦績データ上の症状と差分から導く「試す価値のあるプレー方針」を指す。 |
| 発動条件 | 行動仮説を思い出す場面。例: 「中盤以降、物件収益で上位だが目的地到着がないとき」。 |
| やること | 発動条件に当てはまったとき、優先する行動方針。 |
| 避けること | その仮説において、次回は抑制したい行動方針または見落とし。 |
| データ上の理由 | 行動仮説を出した理由。症状、勝ち切り試合と非勝利試合、または上位試合と下位試合の差分、対象件数、信頼度を含む。 |
| 試合後の検証方法 | 次回開催後に、行動仮説が有効だったかを振り返る観点。 |
| ベースライン | 期間プレイブックの比較基準にする確定済み戦績の集合。原則として画面で選択中の作品、シーズン、マップスコープを使う。 |
| 入賞 | 1位または2位で終えること。既存戦績比較ページの入賞率と同じ定義を使う。 |
| 下位 | 3位または4位で終えること。既存戦績比較ページの下位率と同じ定義を使う。 |
| 勝ち切り | 1位で終えること。 |
| 非勝利試合 | 対象プレーヤーが2位、3位、4位で終えた試合。 |
| 上位試合 | 対象プレーヤーが1位または2位で終えた試合。 |
| 下位試合 | 対象プレーヤーが3位または4位で終えた試合。 |
| 症状 | 本人ベースラインや選択スコープ平均から見て、成績が伸びる条件または崩れる条件。 |
| 差分 | 同じ症状内で、勝ち切り試合と非勝利試合、または上位試合と下位試合の間に出ている違い。 |
| 判断材料 | プレイブックカードに表示する主要データ。指標名、値、対象件数、信頼度を含み、行動仮説の理由をカード内で成立させる。 |

## 3. データ前提と制約

### 3.1 利用できるデータ

以下の既存データだけを使う。

- `matches` の確定済み試合。
- `held_events`、`played_at`、`match_no_in_event`。
- 作品、シーズン、マップ。
- 固定4名のプレー順、順位、総資産、収益額。
- 固定6事件: 目的地、プラス駅、マイナス駅、カード駅、カード売り場、スリの銀次。

### 3.2 利用しない・推定しないデータ

以下はMVPで扱わない。

- 行動順。
- カードの種類、購入、使用内容。
- 目的地までの距離。
- 移動距離、駅数、マス数。
- ターン単位の状況。
- リアルタイムの意思決定。
- 新しいプレーデータ入力。
- AIまたは外部推論エンジン。

### 3.3 現状データで断定しないこと

現状データは試合後の結果と出来事の集計であり、ターンごとの状態やプレーヤーの選択を含まない。そのため、以下は断定しない。

- 特定のカードを買う、使う、温存するべきだったという判断。
- 目的地まで最短で進むべきだったという判断。
- ある物件を買う、独占する、売るべきだったという判断。
- 行動順、ターン数、距離、所持カード、持ち金推移を前提にした局面判断。
- スリの銀次など偶発イベントを、プレーヤーの判断ミスとして扱うこと。

一方で、以下の粒度までは提示してよい。

- 収益で先行したとき、目的地0回のまま終えないように意識する。
- 目的地が取れない展開では、上位狙いより下位回避と総資産維持を優先する。
- 低資産に沈みやすい条件では、逆転狙いを続ける前に資産防衛へ切り替える。
- 苦手番手では、目的地回数と総資産の落ち込みを早めに補正する。
- 銀次被害後は、勝ち切りではなく入賞・下位回避へ方針を切り替える。

発動条件は、アプリが試合中にリアルタイム判定する条件ではない。開催後の振り返りでプレーヤーが持ち帰り、次回の試合中に自己観察するための目安として扱う。

### 3.4 データ量の設計前提

公開仕様には本番DBの実測値を固定値として置かない。設計上は、作品単位では数十〜百数十戦規模の履歴があり、作品全体の傾向は一定の信頼度で扱える前提とする。

ただし、シーズン、マップ、条件付き指標で細かく絞ると対象件数は急に減る。そのため、プレイブックの採用可否は件数と信頼度で制御する。

## 4. IA / UX要求

### 4.1 配置

戦績比較ページの分析サブページタブを以下の並びへ拡張する。

1. 振り返り
2. 順位と相性
3. 勝ち筋
4. 流れと勢い
5. 番手と出来事

「振り返り」は次回4戦で試す行動仮説を決める場所とし、既存4タブは根拠の詳細を読む場所とする。

### 4.2 別ページ化しない理由

振り返りは戦績比較ページの派生ではなく上位解釈レイヤーである。別ページ化すると、作品、シーズン、マップ選択、認証、URL共有、根拠指標への往復が重くなる。MVPでは既存ページ内タブとして実装する。

### 4.3 開催回レビューを含めない理由

開催回単位の要約は、既存の「流れと勢い」タブにある荒れ試合、直近8戦、開催回内第n試合の詳細指標と役割が重なりやすい。また、直近4戦前後だけではプレイブックの推薦根拠として弱く、情報量の薄いセクションになりやすい。

そのため、MVPでは振り返りタブを期間プレイブックに集約する。開催回単位の出来事を確認したい場合は、既存の「流れと勢い」タブを根拠の正本として使う。

### 4.4 画面構造

振り返りタブは、個人別行動プレイブックを中心に構成する。

- プレーヤーごとに最大3件のカードを表示する。
- 「再現する」「見直す」「検証する」の分類を表示する。
- 各カードには「行動仮説」「発動条件」「やること」「避けること」「データ上の理由」「試合後の検証方法」を表示する。
- 詳細リンクを付ける。ただし詳細リンクは追加確認の導線であり、カードの起点は行動仮説とデータ上の理由とする。
- 詳細リンクはカード末尾に置き、独立したリンク集セクションは作らない。
- 複数人に同じカテゴリの候補が出る場合は、個人カードへ同じ内容を繰り返さず、「卓全体で出やすい論点」として最大2件だけ一度表示する。

振り返りタブは、開催回要約、詳細リンク一覧を持たない。画面の主役は、プレーヤー別の行動仮説カードだけにする。共通論点は全体サマリーではなく、同じ候補を各プレーヤー欄へ重複表示しないための選定補助として扱う。

### 4.5 共通情報とカード別情報

一度だけ表示する情報:

- 分析範囲。
- 対象対戦数。
- 「再現する」「見直す」「検証する」の分類の読み方。
- 信頼度表示の読み方。
- 発動条件は次回試合中の自己観察の目安であり、リアルタイム判定ではないこと。
- 複数人に共通して出たカテゴリと、そのカテゴリを個人カードから間引く理由。

プレーヤーごとに表示する情報:

- プレーヤー表示名。
- 行動仮説カード。件数は0〜3件。
- カードが0件の場合の空状態。

カードごとに表示する情報:

- 分類。
- 行動仮説。
- 発動条件。
- やること。
- 避けること。
- データ上の理由。
- 試合後の検証方法。
- 対象件数。
- 信頼度。
- 4人内でどの程度目立つ候補か。
- 詳細リンク。

分析範囲、分類説明、信頼度説明、データ制約の注記はカードごとに繰り返さない。逆に、行動仮説の意味を理解するための症状、差分、対象件数、信頼度は、各カードに必ず表示する。

画面に表示するプレーヤー名は、必ずマスタの表示名を使う。マスタID、APIフィールド名、DB列名、英語の内部キーをUI文言として表示しない。指標名は「物件収益」「番手」「目的地」「スリの銀次」のような日本語ラベルへ変換する。

カードが0件のプレーヤーには、弱い推薦を無理に出さない。空状態では「この条件では、次回行動として出せる強い差分はありません」のように、カードを出さない理由を短く表示する。詳細指標を見れば何か分かるかもしれない、という曖昧な誘導だけで終えない。

### 4.6 情報優先順位

カード内の表示順は以下を原則とする。

1. 行動仮説
2. 発動条件
3. やること
4. 避けること
5. データ上の理由
6. 試合後の検証方法
7. 詳細リンク

プレーヤーは振り返り中に「次の試合で思い出す一文」を持ち帰る必要がある。そのため、数値やリンクを先に見せない。数値は行動仮説の理由として後続に置く。

カードの主見出しは、名詞句ではなく行動方針を含む短文にする。

良い例:

- 収益先行時は目的地0回で終えない。
- 目的地なしの展開では下位回避へ早めに切り替える。
- 銀次被害後は勝ち切りより入賞維持へ切り替える。

悪い例:

- 物件収益トップ時の勝ち方を再現する。
- 目的地なしの普段との差を確認する。
- 銀次被害時の順位スコアを見る。

PCでは4人分を比較しやすい列またはグリッドにする。モバイルでは1人ずつ縦に読む構成にする。どちらの場合も、カード内のラベル順と余白を揃え、カードごとの情報量差でレイアウトが過度にがたつかないようにする。

### 4.7 ライティング / ラベル設計

カード内ラベルは以下を使う。

| ラベル | 役割 |
|---|---|
| 行動仮説 | 試合中に思い出す短い方針。 |
| 発動条件 | その方針を意識する場面。 |
| やること | 優先する行動方針。 |
| 避けること | 次回は抑える行動方針または見落とし。 |
| データ上の理由 | 症状、差分、主要数値。 |
| 試合後の検証 | 次回開催後に振り返る観点。 |

以下のラベルはMVPの振り返りタブでは使わない。

- どう見る
- 期待できる価値
- なぜ
- 補足
- 通常進行
- 今回の要点

理由:

- 「どう見る」「なぜ」「補足」は、根拠説明に寄りやすく、試合中の行動に結びつきにくい。
- 「期待できる価値」は企画説明の語彙であり、画面上のプレーヤー向け文言ではない。
- 「通常進行」「今回の要点」は、開催回レビューと誤認されやすい。

文体は、プレーヤーが試合中に思い出せる短文を優先する。敬体で長く説明するより、行動方針を先に出し、理由は後ろに置く。

### 4.8 URL状態

既存の `view` query に `review` を追加する。

- `view=review` は振り返りタブを表す。
- 振り返り拡張実装後、URL query がない場合は `review` を初期表示にする。
- 既存の直接リンク互換のため、`view=overview`、`view=drivers`、`view=flow`、`view=context` は維持する。
- `view` が不正値の場合は `review` に戻す。
- `reviewHeldEventId` は使用しない。古いURLに含まれる場合も正規化時に落とす。
- 振り返りカード内の詳細リンクは、既存タブへ同一ページ内遷移する。
- 詳細リンクで既存タブへ移動した後にブラウザバックした場合、戦績比較ページから離脱せず、直前の `view=review` 状態へ戻る。
- 作品、シーズン、マップなどの選択状態は、詳細リンク遷移とブラウザバック後も維持する。

| RP-17 | 分析範囲、分類説明、信頼度説明、データ制約の注記は一度だけ表示し、カードごとに繰り返さない。 |
| RP-18 | 詳細リンク一覧セクションは作らない。詳細リンクは各カード末尾の補助導線として表示する。 |
| RP-19 | プレーヤー名、指標名、カテゴリ名は表示用ラベルを使い、マスタID、APIフィールド名、DB列名、英語の内部キーをUI文言として表示しない。 |
| RP-20 | 詳細リンクで既存タブへ移動した後、ブラウザバックで `view=review` の振り返りタブへ戻れる。 |
| RP-21 | 採用できる行動仮説がないプレーヤーには弱いカードを出さず、短い空状態を表示する。 |
| RP-22 | プレイブック候補は本人内差分だけでなく、4人内の相対的な強さを採用スコアに含める。全員に同じカテゴリが出る場合は、特徴が強い人だけを個人カードに残す。 |
| RP-23 | 複数人に共通して出るカテゴリは、最大2件の共通論点として一度だけ表示し、個人カードの重複を避ける。 |

## 6. 既存指標との関係

### 6.1 役割分担

| 領域 | 役割 |
|---|---|
| 振り返り | 既存指標を解釈し、次回4戦で試す行動仮説へ変換する。 |
| 順位と相性 | 平均順位、順位分布、直接対決、順位ブレの詳細根拠。 |
| 勝ち筋 | 資産分布、物件収益、目的地、勝ち筋分類の詳細根拠。 |
| 流れと勢い | 荒れ試合、直近8戦、開催回内第n試合の詳細根拠。 |
| 番手と出来事 | 番手、カード売り場と目的地、スリの銀次の詳細根拠。 |

### 6.2 重複回避

- 振り返りタブでは、詳細指標の表やグラフを再掲しない。
- プレイブックカード内の数値は、行動仮説を理解するために必要な症状と差分に絞る。
- 詳細表やグラフを読みたい場合は既存タブへ移動させる。
- 既存の上部主要サマリーは、振り返り拡張時に縮小を検討する。
- 上部に残す値は、対象スコープ、対戦数、データ信頼度、振り返り導線など、ページ全体の状態把握に寄せる。
- 首位社長、スリの銀次、平均順位差など話題性の強いサマリーは、詳細根拠タブへ寄せる。振り返りタブでは行動仮説に接続できる場合だけ材料として扱う。

### 6.3 既存指標を落とす条件

既存指標は原則として削除しない。削除または移動を検討するのは、以下をすべて満たす場合に限る。

- 振り返りタブと既存タブで同じ対象、同じ分母、同じ表示粒度の値を出している。
- 既存タブ側で詳細根拠としての役割を失っている。
- 削除しても、プレイブックカードの根拠を追跡できる。

## 7. プレイブック生成ルール

### 7.1 基本方針

推薦は決定論的に生成する。AIや自由入力に依存しない。

カードは以下の問いに答える。

- 次回4戦のどんな場面で思い出すべきか。
- その場面で何を優先するのか。
- その場面で何を避けるのか。
- その行動仮説を試す価値はどこにあるのか。
- どのくらい根拠があるか。
- 試合後に何を見れば検証できるか。

カードは「最適手」の断定ではなく、現状データから試す価値がある行動仮説として提示する。

### 7.2 カード分類

| 分類 | 意味 | 例 |
|---|---|---|
| 再現する | すでに成績を支えている行動方針を、次回も意識して再現する。 | 収益先行時に目的地到着も伴って勝てているため、収益先行時は目的地0回で終えない。 |
| 見直す | 成績を落としやすい条件で、次回は優先順位を変える。 | 物件収益トップ時に勝ち切れていないため、追加収益より目的地到着を優先する。 |
| 検証する | 件数が少ない、または差分が中程度のため、次回4戦で試す仮説として扱う。 | 目的地なしでも総資産を残せた試合は入賞できているため、目的地なしの展開で下位回避へ寄せる。 |

### 7.3 候補カテゴリ

MVPの候補カテゴリと行動仮説への変換方針:

| カテゴリ | 症状例 | 行動仮説の方向 |
|---|---|---|
| 物件収益 | 物件収益トップ時に1位率が低い。勝ち切り試合と非勝利試合で目的地差がある。 | 収益先行時は、追加収益だけでなく目的地到着・下位回避を優先する。 |
| 目的地 | 目的地0回時の順位スコアが本人平均より低い。 | 目的地なしの展開では、上位狙いを続けすぎず総資産維持と下位回避へ切り替える。 |
| 低資産 | 選択スコープの低資産帯に入る割合が高い。 | 低資産に沈む前に、逆転狙いから資産防衛・事故回避へ切り替える。 |
| 番手 | 苦手番手で順位スコア、目的地、総資産が落ちる。 | 苦手番手では序盤から目的地回数と総資産の遅れを早めに補正する。 |
| スリの銀次 | 銀次被害時の順位スコアが本人平均より低い。 | 銀次被害後は、勝ち切りより入賞維持・下位回避へ切り替える。 |
| 直近 | 直近8戦の落ち込みが目的地、総資産、銀次などの差分と接続できる。 | 単独カードにはせず、目的地、低資産、銀次など具体カテゴリの採用優先度を上げる材料にする。 |
| カード売り場と目的地 | 売り場あり・到着なしの試合で成績が落ちる、または到着あり・売り場ありで成績が伸びる。 | 単独カードにはせず、目的地や低資産カードのデータ上の理由に補助差分として使う。 |

直近、カード売り場と目的地、相性、直接対決、順位ブレは、MVPでは単独の行動仮説にしない。具体的な試合中行動へ接続できる補助差分がある場合だけ、主カテゴリの理由として使う。

### 7.4 順位スコア

順位系の比較では、必要に応じて順位スコアを使う。

```text
rankScore = 5 - rank
```

1位=4点、2位=3点、3位=2点、4位=1点とする。高いほど良い。

### 7.5 生成パイプライン

プレーヤーごとに以下の順で行動仮説候補を作る。

1. 症状を検出する。
2. 症状に該当する試合を、勝ち切り試合と非勝利試合、または上位試合と下位試合へ分ける。
3. 目的地、総資産、物件収益、スリの銀次、カード売り場、番手、第n試合など、現状データで比較できる差分を計算する。
4. 差分が行動へ接続できる場合だけ、行動テンプレートへ変換する。
5. プレーヤー間で同カテゴリ候補の強さを比較し、4人内の特徴度を計算する。
6. 採用スコアを計算する。
7. 同カテゴリの重複と共通化しすぎた候補を除外し、最大3件に絞る。
8. カード本文をテンプレートで生成する。

### 7.6 症状スコア

候補ごとに、本人ベースラインまたは選択スコープ全体との差分を計算する。

例:

- 条件あり平均順位スコア - 本人平均順位スコア。
- 条件あり入賞率 - 本人入賞率。
- 条件あり下位率 - 本人下位率。
- 直近8戦平均順位スコア - 長期平均順位スコア。
- 物件収益トップ時1位率 - 本人全体1位率。

症状スコアは表示上の断定に使う前に、対象件数と信頼度で調整する。

### 7.7 差分スコア

症状だけでは行動仮説にしない。症状に該当する試合の中で、勝ち切り試合と非勝利試合、または上位試合と下位試合の差分を見る。

比較対象:

- 目的地平均。
- 総資産平均。
- 物件収益平均または物件収益比率。
- スリの銀次平均。
- カード売り場回数。
- 番手。
- 開催回内第n試合。

MVPでは、平均差または標準化差分を使ってよい。標準化差分は、同一カテゴリ内で0〜1へ正規化して `contrastScore` に変換する。分散が極端に小さい場合は、ゼロ除算や過大評価を避けるため、そのカテゴリの差分スコアを下げる。

件数が少ない条件や外れ値の影響が大きい条件では、Cliff's delta のような順位ベースの効果量を後続改善候補とする。MVPでは必須ではないが、平均差だけで説明しづらいカテゴリに導入しやすいよう、差分計算はカテゴリ別関数として分ける。

差分が小さい、または行動へ接続できない場合はカード化しない。例えば「銀次被害時に順位が悪い」だけではカード化せず、被害後に総資産や目的地の残り方で差が見える場合に限定する。

### 7.8 縮小推定

対象件数が少ない条件の極端な値を過大評価しないため、MVPでは単純な縮小推定を使ってよい。

順位スコア差の例:

```text
shrunkEffect = rawEffect * targetCount / (targetCount + priorWeight)
```

`priorWeight` は固定閾値として実装に集約する。初期値は8を目安にしてよいが、テストで固定し、将来調整可能にする。

率の例:

```text
shrunkRate = (successCount + priorWeight * baselineRate) / (targetCount + priorWeight)
```

率の信頼度表示には、Wilson区間または Beta-Binomial の考え方を後続で採用してよい。MVPでは縮小率と件数閾値を優先する。

### 7.9 信頼度

カード候補は以下の件数基準で扱う。

| 対象 | 主採用 | 参考 | 非表示 |
|---|---:|---:|---:|
| 通常指標 | 20戦以上 | 3〜19戦 | 2戦以下 |
| 条件付き指標 | 8件以上 | 3〜7件 | 2件以下 |

通常指標は平均順位、入賞率、順位ブレ、資産分布など、プレーヤーの全参加試合を分母にする指標を指す。

条件付き指標は、物件収益トップ時、目的地ゼロ時、スリの銀次被害時、特定番手、特定開催回内試合番号など、条件で対象を絞る指標を指す。

### 7.10 行動接続度

行動接続度は、その差分を試合中の意識へ変換しやすいかを表す。主観で都度決めず、カテゴリごとの定数として扱う。

| 行動接続度 | 対象 | 理由 |
|---|---|---|
| 高 | 目的地、低資産回避、収益先行時の目的地到着、総資産維持 | プレーヤーが試合中に優先順位として意識しやすい。 |
| 中 | 番手、銀次被害後の切り替え | 直接制御しにくいが、方針変更として意識できる。 |
| 低 | 直近順位、相性、カード売り場と目的地、偶発イベント単体 | 行動へ接続しにくい。単独では主行動仮説にしない。 |

### 7.11 採用スコア

カード候補は以下の考え方で優先順位を付ける。

```text
actionAdviceScore =
  abs(shrunkSymptomScore)
  * contrastScore
  * exposureWeight
  * reliability
  * actionConnection
  * peerDistinctiveness
  * commonnessPenalty
```

| 要素 | 意味 |
|---|---|
| `shrunkSymptomScore` | 縮小後の症状スコア。大きいほど成績差が目立つ。 |
| `contrastScore` | 勝ち切り試合と非勝利試合、または上位試合と下位試合の差分の明確さ。 |
| `exposureWeight` | その条件が発生する頻度。頻度が高いほど次回も試しやすい。 |
| `reliability` | 件数と信頼度。件数不足なら下げる。 |
| `actionConnection` | 試合中の行動へ変換しやすい度合い。 |
| `peerDistinctiveness` | 同じカテゴリの候補内で、そのプレーヤーの症状または差分が4人内でどれだけ目立つか。4人固定のため、z-scoreより順位、中央値差、分位を優先する。 |
| `commonnessPenalty` | 同じカテゴリが3人以上に出る場合、相対的に弱い候補を個人カードから外すためのペナルティ。 |

4人内相対評価は、候補カテゴリごとに効果量の絶対値またはカテゴリ固有の悪化率で順位付けする。1位を最も強く、2位を次点、3位以下は共通論点側へ寄せる。サンプルが少ないスコープでは、信頼度と縮小推定を優先し、相対順位だけで強く断定しない。

### 7.12 カード選定

プレーヤーごとに以下の順で選ぶ。

1. 候補カテゴリごとに最も強いカード候補を作る。
2. 全プレーヤーの同カテゴリ候補を集め、4人内の特徴度を計算する。
3. 非表示条件に該当する候補を除外する。
4. 同じカテゴリが3人以上に出る場合、上位1〜2人程度だけを個人カードに残し、残りは共通論点へ寄せる。
5. `actionAdviceScore` で並べる。
6. 同じカテゴリから複数出さない。
7. 可能なら「再現する」「見直す」「検証する」が偏りすぎないようにする。
8. 最大3件に絞る。
9. 参考カードだけになる場合は、強い断定表現を避ける。
10. 行動接続度が低い候補だけの場合は、カードを出さない。

### 7.13 行動テンプレート

カード本文はカテゴリ別テンプレートで生成する。AIによる自由文生成はMVPでは使わない。

| カテゴリ | 行動仮説テンプレート | 発動条件テンプレート | やることテンプレート | 避けることテンプレート |
|---|---|---|---|---|
| 物件収益 | 収益先行時は目的地0回で終えない。 | 中盤以降、物件収益で上位だが目的地到着がないとき。 | 追加収益より、目的地周辺への位置取り、到着、下位回避を優先する。 | 収益トップだから安全と見て、目的地0回のまま終盤へ入ること。 |
| 目的地 | 目的地なしの展開では下位回避へ早めに切り替える。 | 目的地到着がないまま中盤を過ぎ、総資産も伸びていないとき。 | 一発逆転より、総資産を残す動きと事故回避を優先する。 | 目的地を取れないまま、資産も削る展開を続けること。 |
| 低資産 | 低資産に沈む前に資産防衛へ切り替える。 | 総資産が伸びず、目的地や収益でも遅れていると感じるとき。 | 逆転狙いを続ける前に、資産を残す選択と事故回避を優先する。 | 低資産のまま大きな上振れだけを待つこと。 |
| 番手 | 苦手番手では序盤の遅れを早めに補正する。 | 苦手番手に入り、目的地回数や総資産が伸びないとき。 | 目的地到着が遅れているなら目的地優先、総資産が伸びていないなら資産防衛へ早めに寄せる。 | 番手差を無視して普段通りの優先順位で進め続けること。 |
| スリの銀次 | 銀次被害後は勝ち切りより入賞維持へ切り替える。 | スリの銀次被害を受け、総資産差が広がったとき。 | 1位狙いを続けるかではなく、入賞・下位回避に必要な資産維持を優先する。 | 被害前と同じ勝ち切り方に固執すること。 |

テンプレートは実データの差分に応じて出し分ける。例えば、物件収益トップ時の非勝利試合で目的地差が大きい場合は目的地優先の仮説にし、銀次差が大きい場合は事故後の方針切り替えに寄せる。

### 7.14 表現ルール

- 「勝てる」「負ける」と断定しない。
- 主見出しは、プレーヤーが試合中に思い出せる短い行動文にする。
- 「確認する」「見る」「切り分ける」は試合後の検証方法にだけ使う。行動仮説ややることの主文には使わない。
- 「傾向がある」「可能性がある」はデータ上の理由で使う。行動仮説は「〜を優先する」「〜で終えない」「〜へ切り替える」のように具体化する。
- 行動順やカード内容が必要な因果は書かない。
- 個人攻撃、能力断定、煽り表現を避ける。
- ネタ表現は見出しに限定してよいが、根拠行は堅く書く。

### 7.15 カード文例

カードは、抽象的な提案の後にリンクだけを置くのではなく、判断に使った差分を本文内で示してから行動仮説へ接続する。

物件収益カテゴリの例:

| 項目 | 文例 |
|---|---|
| 行動仮説 | 収益先行時は目的地0回で終えない。 |
| 発動条件 | 中盤以降、物件収益で上位だが目的地到着がないとき。 |
| やること | 追加収益より、目的地周辺への位置取り、到着、下位回避を優先する。 |
| 避けること | 収益トップだから安全と見て、目的地0回のまま終盤へ入ること。 |
| データ上の理由 | 収益上位の試合のうち、上位試合は目的地平均 `{upperDestinationAvg}` 回、下位試合は `{lowerDestinationAvg}` 回。下位試合では目的地0回が `{zeroDestinationLossCount}` 件あり、収益だけでは順位差を埋めきれていない可能性がある。 |
| 試合後の検証 | 次回、収益で上位だった試合を対象に、目的地0回で終えたか、入賞できたかを振り返る。 |

目的地カテゴリの例:

| 項目 | 文例 |
|---|---|
| 行動仮説 | 目的地なしの展開では下位回避へ早めに切り替える。 |
| 発動条件 | 目的地到着がないまま中盤を過ぎ、総資産も伸びていないとき。 |
| やること | 一発逆転より、総資産を残す動きと事故回避を優先する。 |
| 避けること | 目的地を取れないまま、資産も削る展開を続けること。 |
| データ上の理由 | 目的地0回の試合は `{targetCount}` 件あり、順位スコアが本人平均より `{rankScoreDiff}` 低い。上位試合の総資産平均は `{upperTotalAssetsAvg}`、下位試合は `{lowerTotalAssetsAvg}` で、目的地がない展開では資産維持が入賞の分岐になっている可能性がある。 |
| 試合後の検証 | 次回、目的地0回だった試合を対象に、総資産を残せたか、4位を避けられたかを振り返る。 |

上記の文例は固定文そのものではなく、最低限の構成を示す。実装では対象件数、平均差、信頼度を文中または隣接する数値行で補う。

## 8. 詳細根拠との往復

### 8.1 役割

振り返りタブは、詳細表やグラフを再掲しない。カードごとに、行動仮説の判断に使った症状、差分、主要データを表示し、詳細タブの該当ブロックへ移動できる導線を持たせる。

詳細タブは、カード内の読み取りを深掘りする場所である。振り返りタブ上で何を試すべきかが分からない状態のまま、利用者を詳細タブへ移動させない。

開催回単位の出来事、荒れ試合、直近の流れ、第n試合別の傾向は「流れと勢い」タブで確認する。振り返りタブには開催回セレクトや開催回要約を置かない。

### 8.2 詳細リンク

プレイブックカードは以下のいずれかの詳細ブロックへリンクする。

- 順位と相性: 平均順位、順位分布、直接対決、順位ブレ。
- 勝ち筋: 資産分布、物件収益、目的地。
- 流れと勢い: 荒れ試合、直近8戦、開催回内第n試合。
- 番手と出来事: 番手、カード売り場、スリの銀次。

### 8.3 禁止事項

- 詳細タブと同じ表やグラフを振り返りタブへ再掲しない。
- 判断材料を詳細リンクだけに置き、カード内では抽象的な説明だけにしない。
- 行動仮説を「確認する」「見る」「切り分ける」だけで終わらせない。
- 開催回単位の薄い要約を、行動仮説と同列に置かない。
- 全事件簿合算を総行動数、総移動量、総駅数として扱わない。

## 9. API / 実装方針

### 9.1 API単位

MVPでは、既存の選択肢APIと集計APIを維持しつつ、単一の振り返りAPIを追加する。

```text
GET /api/analytics/series-comparison/review?gameTitleId=...&seasonMasterId=...&mapMasterId=...
```

理由:

- 既存集計レスポンスを過度に肥大化させない。
- 詳細指標全体を過度に肥大化させず、プレイブック生成結果を独立して取得できる。
- 指標別APIを乱立させず、振り返りビュー単位の1 APIに閉じられる。

### 9.2 レスポンスの意味単位

振り返りレスポンスは、UIカード構造そのものではなく、以下の意味単位で返す。

- `baseline`: 比較基準スコープ、対象件数、補助ベースライン有無。
- `commonPlaybookTopics`: 複数人に共通して出たカテゴリの短い論点。個人カードの重複を避けるための補助であり、詳細サマリーではない。
- `playbookByPlayer`: プレーヤー別の行動仮説カード候補。
- `actionHypothesis`: 試合中に思い出す短い行動方針。
- `triggerCondition`: 行動仮説を発動する場面。
- `recommendedAction`: 発動条件に当てはまったとき優先する行動方針。
- `avoidAction`: 次回は避けたい行動方針または見落とし。
- `dataReason`: 症状、差分、主要数値をまとめた理由。
- `postMatchCheck`: 次回開催後に検証する観点。
- `evidence`: カードごとの判断材料。指標名、値、対象件数、信頼度を含み、カード内の理由を支える。
- `dataQuality`: サンプル不足、対象なし、同値あり、除外理由。

### 9.3 実装境界

- 推薦生成ロジックはAPI側に置くことを推奨する。
- webは表示用ViewModel、詳細リンク、空状態、エラー表示に集中する。
- 推薦生成の閾値、`priorWeight`、`actionConnection`、行動テンプレートはAPI側定数として集約し、テストで固定する。
- MVPの計算は標準ライブラリで実装できる範囲に留める。Wilson区間、Beta-Binomial、Cliff's delta などを本採用する段階で、専用ライブラリ導入を検討する。
- DBスキーマ変更はMVPでは不要とする。
- 対戦数増加で遅くなった場合だけ、キャッシュや集計テーブルを検討する。

## 10. 非機能要求

| 観点 | 要求 |
|---|---|
| 認証・権限 | 既存戦績比較ページと同じく、ログイン済みの有効アカウントだけが閲覧できる。 |
| セキュリティ | 参照専用APIとし、既存のCookie認証、dev/test account header本番無効化、Problem Details正規化に従う。 |
| データ正確性 | 順位は保存済み `rank` を正とし、総資産から再計算しない。 |
| 説明責任 | すべての行動仮説に、症状、差分、主要な判断材料、対象件数、信頼度、詳細リンクを付ける。 |
| 低コスト運用 | MVPでは外部推論エンジンを呼ばないため追加API課金は発生しない。 |
| 性能 | 固定4名、作品単位で数十〜百数十戦規模のデータを快適に表示できる。 |
| エラー表示 | レビューAPIだけが失敗しても、既存の戦績比較詳細タブは閲覧できる。 |
| アクセシビリティ | カード分類、信頼度、リンク先は色だけで意味を伝えない。 |
| レスポンシブ | PCでは4人のプレイブックを比較しやすく、モバイルでは1人ずつ読みやすい構成にする。 |

## 11. MVP範囲外

MVPでは以下を扱わない。

- AIによる自然文生成。
- Discord投稿文の自動生成。
- プレーヤー別行動仮説の保存。
- 次回開催後の行動仮説達成判定。
- 複雑な予測モデル。
- カード内容、行動順、目的地距離など新規プレーデータの取得。
- 共有用画像生成。

将来AIを使う場合も、AIに推薦対象を決めさせない。決定論的に選ばれた行動仮説、症状、差分、根拠を自然文へ整える用途に限定する。

## 12. テスト要求

実装時は、少なくとも以下を固定する。

- `view=review` のURL復元。
- 古い `reviewHeldEventId` query が正規化時に落ちること。
- 詳細リンク遷移後、ブラウザバックで戦績比較ページを離脱せず `view=review` に戻ること。
- 詳細リンク遷移とブラウザバック後に、作品、シーズン、マップの選択状態が維持されること。
- ベースライン件数不足時の補助ベースライン表示。
- 通常指標、条件付き指標の主採用、参考、非表示の境界。
- `rankScore`、症状スコア、差分スコア、縮小推定、`actionAdviceScore` の計算。
- プレーヤーごとの最大3件制限。
- 採用候補がないプレーヤーの空状態表示。
- 同じカテゴリから複数カードを出さないこと。
- 行動仮説、発動条件、やること、避けること、データ上の理由、試合後の検証方法の表示。
- 判断材料の対象件数と信頼度表示。
- 詳細リンクだけに依存せず、物件収益トップ時の勝敗差など行動仮説判断に使う補助データがカード内に出ること。
- 行動仮説とやることが「確認する」「見る」「切り分ける」だけで終わらないこと。
- 未取得データを前提にしたカード選択、物件選択、距離判断が表示されないこと。
- プレーヤーのマスタID、APIフィールド名、DB列名、英語の内部キーがUIに表示されないこと。
- 既存タブへの詳細リンク。
- レビューAPI失敗時に既存集計表示を壊さないこと。

`docs/post-mortem/lessons.md` では、実装時に L1、L2、L3、L5、L7 が該当する。
