# ドメインルール

業務要件の正本は `requirements/base.md` を参照する。このファイルは実装者向けのモデル・用語・フローの規約を定める。

## 1. 用語定義

| 用語 | 定義 |
|---|---|
| 開催回 (`held_event`) | 1夜の桃鉄会。複数の試合を含む場合がある。 |
| 試合 (`match`) | 桃鉄1年勝負の1回分。1つの `held_event` に属する。 |
| 試合参加者 (`match_player`) | 1試合における1名のプレーヤーと結果（資産・順位・プレー順）。 |
| プレーヤー (`member`) | 固定4名のメンバー。Discord IDで識別する。 |
| エイリアス (`member_alias`) | OCR結果のプレーヤー名表記を正式メンバーに紐づけるマッピング。 |
| OCRドラフト (`ocr_draft`) | OCR解析結果の下書き。確定前に手修正できる。 |
| OCRジョブ (`ocr_job`) | 画像アップロードからOCR完了までの非同期ジョブ。 |

## 2. 試合確定の必須条件

試合結果の確定には以下がすべて必要。

- 開催日時
- ゲームタイトル
- シーズン
- 親（オーナー）
- マップ
- 全4名のプレー順（重複なし）
- 全4名のランク（1〜4の重複なし整数）

**ランクはOCRまたは手修正で直接入力する。資産額から計算しない。**

## 3. 数値表現

- 資産・収益は整数で保存し、単位は万円とする（例: `10000` = 1億円）。
- 表示フォーマットはUI側の責任。

## 4. OCRドラフトから確定までのフロー

```text
画像アップロード
  → OCRジョブ作成（DB記録 + Redisキュー投入）
  → OCRワーカーが処理
  → OCRドラフト保存（DB）
  → ユーザーがドラフトを手修正
  → 確定（match / match_players へ書き込み）
  → 画像削除
```

- 画像はOCR完了後に削除する。恒久保存しない。
- ドラフトは確定前であれば何度でも上書き可能。
- OCR失敗はジョブ失敗として記録し、再アップロードを促す。
