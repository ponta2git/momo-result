# 桃鉄の結果を記録し、集計するアプリケーションの要求仕様

毎週開催している深夜の桃鉄1年勝負の結果を記録し、後からCSV/TSVとして出力できるようにする。

本書は業務要求とMVP範囲の正本である。実装詳細、技術選定、検証コマンド、運用手順は `docs/requirements/system-design.md`、`docs/architecture.md`、`docs/domain-rule.md`、`docs/db-rule.md` を参照する。

---

## 1. 概要

- 操作はWeb上で行う。
- 仲間内だけで運用する。
- ログイン可能な利用者は `momo_login_accounts` で明示的に許可する。
- 試合参加者は summit アプリと共有する固定4名の `members` とする。
- ログインアカウントと試合参加者は別概念とする。`momo_login_accounts.player_member_id` がある場合だけ、ログインアカウントと `members` が紐づく。
- MVP では、画像取り込み、OCR下書き作成、手修正、確定記録、確定済み記録の編集・削除、CSV/TSV出力までを対象とする。
- 集計画面などの参照系UIは後から同じアプリケーション内に実装する。MVPでは参照画面よりもCSV/TSV出力を優先する。

---

## 2. summit アプリ・momo-db との連携

同じ開発ツリーにある summit アプリと DB 管理リポジトリ（`../momo-db`）と連携する。

- DBスキーマと migration の正本は momo-db リポジトリに集約する。
- summit アプリと本アプリは同じ Neon PostgreSQL スキーマを共有する。
- momo-db の `members`、`held_events`、`held_event_participants`、`app_sessions` などを契約として参照する。
- summit アプリの以下の情報を共有する。
  - 固定4名のメンバー情報（`members`）
  - 実開催履歴（`held_events`）。summit が作成した held_event は Discord 出席 session に紐づく。
- 本アプリ側でも開催履歴を作成できる。本アプリ作成分は summit の出席 session に紐づかない（`held_events.session_id` が NULL）。
- 1つの `held_events` には、第1試合、第2試合のように複数の桃鉄1年勝負の試合結果を紐づけられる。
- 試合番号は同じ `held_events` 内で採番し、必要に応じて手動変更できる。同じ開催回内で同じ試合番号を重複させない。
- 結果作成時は開催履歴を選択し、その中に第n試合を追加する。開催履歴が存在しない場合は、本アプリから新規作成できる。

## 3. 認証・権限

### 3.1 ログイン

- ログイン方式は Discord OAuth とする。
- Discord OAuth のユーザーは `momo_login_accounts.discord_user_id` と照合する。
- `login_enabled = false` のアカウントはログインできない。無効化されたアカウントの既存 session は無効化対象とする。
- `player_member_id = NULL` のログインアカウントは、戦績を操作できるが試合参加者ではない運用者として扱う。

### 3.2 管理者

- 管理者は `momo_login_accounts.is_admin` で付与する。
- 初期管理者は「ぽんた」とする。
- 管理者だけがログイン可能アカウントの許可を管理できる。
- 最低1名の有効な管理者を維持する。

### 3.3 操作権限

ログイン済みの有効アカウントは、誰でも以下を実行できる。

- 開催履歴の閲覧・作成
- 紐づく試合結果・下書きがない開催履歴の削除
- 画像アップロードとOCRジョブ作成
- 自分が作成した未確定下書きの更新・削除
- 試合結果の作成、確定、修正
- 確定済み試合結果の削除
- CSV/TSVダウンロード

管理者だけが以下を実行できる。

- ログイン可能アカウントの作成・更新
- 作品、マップ、シーズン、プレーヤー名エイリアスの作成・更新・削除

MVPでは事件名マスタは固定6項目を読み取り専用で扱う。

確定済み結果の修正履歴は保存しない。

## 5. 画像取り込み

### 5.1 取り込み方法

以下の両方に対応する。

- USB Video Class対応キャプチャーボードからのブラウザ撮影
- 画像ファイルアップロード

キャプチャーボード撮影UIには以下を含める。

- プレビュー
- 静止画撮影
- 再撮影
- 画像アップロードへの切替

対象ブラウザは Chrome / Firefox / Safari / Edge の最新安定版とする。ブラウザや端末の都合でキャプチャーボード撮影が使えない場合は、画像アップロードへフォールバックできればよい。

### 5.2 OCR対象画面種別

取り込む画像のOCR対象画面種別は以下の3種類である。

- 総資産
- 収益額
- 桃鉄事件簿

総資産と収益額の画像には、それぞれのプレーヤーと金額が順位順に表示されている。

桃鉄事件簿の画像には、各プレーヤーが遭遇したイベントの回数が表示されている。

MVPのUIでは、画像を3種類の分類トレイへ配置して読み取り対象を決める。分類が違う場合はユーザーが手動で入れ替えられる。API/OCR境界では `requestedScreenType` と `detectedScreenType` として扱い、「画像種別」ではなく「OCR対象画面種別」として表現する。

同じ種類の画像を複数取り込んだ場合は、最新の取り込み結果で上書きする。

### 5.3 画像の保存

- 画像は結果確定の必須条件ではない。
- 画像なしの手入力だけでも結果を確定できる。
- アップロード可能な画像形式は PNG/JPEG/WebP とする。
- アップロード画像は1枚3MBまでとする。
- API の upload / OCR 公開レスポンスは内部ファイルパスを返さない。画像参照は `imageId` と下書き単位の source image API で扱う。
- 取り込んだ元画像は、下書き確定またはキャンセルまでプレビュー・ダウンロード可能にする。
- 下書き確定またはキャンセル後、元画像は削除対象とする。
- DBには画像実体を残さない。DBに残すのは source image ID、保持期限、削除時刻などのメタデータだけとする。
- サーバーに画像を恒久保存しない。

## 6. OCR・下書き

### 6.1 OCR処理

- OCR/画像解析には外部APIを使わず、ローカル処理に限定する。
- OCRはサーバー側の OCR worker で実行し、ブラウザは画像を一時アップロードする。
- バックエンドはAPIサーバーとOCRワーカーを分離する。
- OCRジョブ状態の正本はDBに置く。
- OCRジョブ配送には Redis Streams を使う。
- API はOCRジョブ作成時にDBへ durable enqueue intent を保存し、Redis publish は outbox dispatcher が行う。
- OCR処理は、将来差し替え可能な構造にする。

### 6.2 下書き

- OCR結果は `ocr_drafts` と `match_drafts` を通じて下書きとして保存する。
- 下書きはユーザーが修正・確認できる。
- 未確定下書きは作業中の間だけDB上に残る。作成者は削除でき、削除時は元画像保持を閉じた上で `match_drafts` を物理削除する。
- `confirmed` は確定 usecase の副作用とセットでだけ到達する。`cancelled` は過去データ互換の終端状態であり、新規のユーザー削除では残さない。
- OCRが失敗した場合は失敗理由を表示し、空の手入力下書きとして続行できる導線を残す。
- OCR結果の信頼度が低そうな項目や警告は、確認画面で強調表示する。

## 7. 結果記録

### 7.1 確定時の必須項目

試合結果の確定には以下を必須とする。

- 開催回
- 開催日時
- 開催回内の試合番号
- 作品名
- シーズン
- オーナー
- マップ
- 固定4名全員のプレー順
- 固定4名全員の順位

開催日時は `held_events` にあるものを初期値として使える。ただし、手編集可能とする。

開催回内の試合番号は1以上とし、同じ開催回内で重複させない。

固定4名全員の順位は 1〜4 で重複なく入力されている必要がある。

### 7.2 プレイヤー別結果

プレイヤーごとに以下を記録する。

- メンバーID
- プレー順
- 順位
- 総資産
- 収益額
- 事件簿の各項目の回数

プレイヤーは固定4名全員を1回ずつ含む必要がある。

総資産額と収益額は、万円単位の整数に正規化して保存する。

順位は総資産額から自動計算せず、OCRまたは手入力された順位を正として保存する。

プレー順は、画像内の色順から判別する。色順は青、赤、黄、緑の順で、画面下のインジケータ左から順番になっている。判別結果は手修正可能とする。

OCR上のプレーヤー名とDB内のメンバー名またはDiscord名が一致しない場合があるため、メンバーごとに複数の表示名エイリアスを持てるようにする。

### 7.3 オーナー・マップ・シーズン

- オーナーは固定4名のいずれかとする。
- 作品にはマップ候補とシーズン候補を紐づける。
- 試合結果ごとに作品、シーズン、オーナー、マップを入力する。
- 選択したマップとシーズンは、選択した作品に属している必要がある。

### 7.4 試合確定の2経路

試合確定には同じ `matches` 作成でも2つの経路がある。

| 経路 | 識別子 | 必須の副作用 |
|---|---|---|
| 手入力 / 画像なしの直接作成 | `matchDraftId` なし | `matches` / `match_players` / `match_incidents` を作成する。`match_drafts` は触らない。 |
| OCR下書きからの確定 | `matchDraftId` あり | `matches` / `match_players` / `match_incidents` を作成し、元 `match_drafts` を `confirmed` にして `confirmed_match_id` を保存し、元画像保持を閉じる。 |

`draftIds` は確定済み試合が参照した OCR 結果の履歴であり、元作業単位を閉じる識別子ではない。OCR下書きから確定する経路では、`draftIds` だけでなく `matchDraftId` を必ず保持して API に渡す。

## 8. マスタ

MVPでは以下のマスタを扱う。

- 作品
- マップ
- シーズン
- 事件名
- プレーヤー名エイリアス

### 8.1 作品・マップ

- 作品はマスタとして登録する。
- マップは作品に紐づける。
- 作品・マップの作成、更新、削除は管理者だけが行える。
- 新規作成時の表示順はAPI側で末尾追加として採番する。

### 8.2 シーズン

- シーズンは期間で区切る大会・集計単位である。
- 1つのシーズンは複数の開催結果を束ねる。
- シーズンは作品に紐づける。
- 結果確定時はシーズン選択を必須とする。
- シーズンの作成、更新、削除は管理者だけが行える。
- 新規作成時の表示順はAPI側で作品ごとの末尾追加として採番する。

### 8.3 事件名

桃鉄事件簿は、事件名マスタとプレイヤーごとの回数として構造化して記録する。

MVPでは以下の6項目を固定で扱い、アプリ上では読み取り専用とする。

- 目的地
- プラス駅
- マイナス駅
- カード駅
- カード売り場
- スリの銀次

### 8.4 プレーヤー名エイリアス

- OCR名寄せ用に、メンバーごとの表示名エイリアスを管理する。
- エイリアスの作成、更新、削除は管理者だけが行える。
- OCR実行時は、入力された hints とDB上のエイリアスを合わせて worker に渡せる。

## 9. CSV/TSV出力

### 9.1 権限

- CSV/TSVダウンロードは非公開APIとする。
- ログイン中UIからのみ実行できる。
- ログイン済みの有効アカウントがダウンロードできる。

### 9.2 出力範囲

以下の範囲を指定してCSV/TSVを出力できる。

- 全試合
- シーズン単位
- `held_events` 単位
- 試合単位

`format` は `csv` または `tsv` とする。

### 9.3 行構造

CSV/TSVは、1行を1プレイヤーの試合結果明細とする。

列は以下の順序を正とする。

| 列 | 内容 |
|---|---|
| シーズン | シーズン名 |
| シーズンNo. | シーズン内の対戦連番 |
| オーナー | 試合結果ごとに入力されたオーナー |
| マップ | 試合結果ごとに入力されたマップ |
| 対戦日 | 開催日 |
| 対戦No. | 同じ作品内の通算対戦番号 |
| プレー順 | プレイヤーのプレー順 |
| プレーヤー名 | プレイヤー名 |
| 順位 | 最終順位 |
| 総資産 | 万円単位の総資産 |
| 収益 | 万円単位の収益額 |
| 目的地 | 桃鉄事件簿の回数 |
| プラス駅 | 桃鉄事件簿の回数 |
| マイナス駅 | 桃鉄事件簿の回数 |
| カード駅 | 桃鉄事件簿の回数 |
| カード売り場 | 桃鉄事件簿の回数 |
| スリの銀次 | 桃鉄事件簿の回数 |

`シーズンNo.` は選択範囲内だけでなく該当シーズン全体での連番とする。`対戦No.` は選択範囲内だけでなく該当作品全体での通算対戦番号とする。

## 10. 技術・構成

- フロントエンドは React/Vite のSPAとする。
- バックエンドAPIとフロントエンドは分離する。
- バックエンドは APIサーバー + OCRワーカーを同一リポジトリ、同一Fly.io構成内で分離する。
- バックエンドは Fly.io を使う。
- DBは Neon を使い、summit アプリと共有する。
- OCRジョブキューには Upstash Redis Streams を使う。
